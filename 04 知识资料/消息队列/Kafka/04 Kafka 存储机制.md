# Kafka 存储机制

## 一句话结论

Kafka 以 Partition 为目录、Segment 为文件单元组织磁盘存储，通过稀疏索引加速消息查找，支持按时间和大小两种保留策略，利用顺序写和分段设计实现高效的文件管理。

## 文件存储结构

### 整体结构

```
Topic (orderMq)
├── Partition 0 (orderMq-0/)
│   ├── 00000000000000000000.index    ← Segment 索引文件
│   ├── 00000000000000000000.log      ← Segment 数据文件
│   ├── 00000000000000368769.index
│   ├── 00000000000000368769.log
│   └── ...
├── Partition 1 (orderMq-1/)
│   └── ...
└── Partition 2 (orderMq-2/)
    └── ...
```

| 层级 | 说明 |
|------|------|
| **Topic** | 逻辑概念，对应多个 Partition 目录 |
| **Partition 目录** | 命名规则：`Topic名称-序号`（从 0 开始） |
| **Segment 文件** | 每个 Partition 由多个 Segment 组成，成对出现（.index + .log） |

### Segment 文件

| 文件类型 | 后缀 | 内容 |
|----------|------|------|
| **索引文件** | `.index` | 存储元数据，指向数据文件中 message 的物理偏移地址 |
| **数据文件** | `.log` | 存储实际消息数据 |

**命名规则**：以该 Segment 第一条消息的 Offset（19 位数字，不足补 0）命名。

> 例如 `00000000000000368769.index` 表示该 Segment 起始 Offset 为 368769。

### 为什么分段？

| 好处 | 说明 |
|------|------|
| 便于清理 | 直接删除整个 Segment 文件，无需逐条删除 |
| 快速定位 | 二分查找文件名即可定位到目标 Segment |
| 减少索引大小 | 每个 Segment 独立索引，避免全局大索引 |

## 稀疏索引

### 设计原理

Kafka 的索引不是为每条消息建立索引，而是**每隔一定数量的消息建立一个索引点**：

| 特点 | 说明 |
|------|------|
| 稀疏存储 | 减少索引文件大小，可全部加载到内存 |
| 二分查找 | 在索引文件中二分查找最近的索引点 |
| 顺序扫描 | 从最近索引点开始顺序扫描找到目标消息 |

### 消息查找流程

以查找 `offset=368776` 为例：

**第一步：定位 Segment 文件**

| Segment 文件 | 起始 Offset |
|-------------|------------|
| `00000000000000000000` | 0 |
| `00000000000000368769` | 368769 |
| `00000000000000737337` | 737338 |

二分查找 → `368776` 落在 `00000000000000368769` 对应的 Segment。

**第二步：通过 Segment 查找消息**

1. 通过 `.index` 索引文件定位最近索引点的物理位置
2. 在 `.log` 数据文件中从该位置顺序扫描到 `offset=368776`

## 数据保留策略

Kafka 支持两种数据保留策略，**满足任一条件即清理**：

| 策略 | 配置参数 | 说明 |
|------|---------|------|
| **按时间** | `log.retention.hours` | 超过指定时间的消息被清理 |
| **按大小** | `log.retention.bytes` | 分区日志超过指定大小时清理旧数据 |

> 例如：同时设置 7 天和 10G，第 5 天达到 10G 时就会触发清理。

## 消息格式

### 消息结构

| 部分 | 大小 | 说明 |
|------|------|------|
| Offset | 8 bytes | 消息在 Partition 中的偏移量 |
| 消息长度 | 4 bytes | 消息总长度 |
| 版本号 (magic) | 1 byte | 文件格式版本 |
| CRC32 | 4 bytes | 校验码，判断消息体是否正常 |
| Attributes | 1 byte（magic=1 时） | 压缩格式等属性 |
| 消息体 (body) | N bytes | 具体的 Key/Value 消息 |

## 常见问题

### Q1：Kafka 如何查找指定 Offset 的消息？

1. 根据 Offset 二分查找定位到对应的 Segment 文件（文件名 = 起始 Offset）
2. 在 Segment 的 `.index` 索引文件中二分查找最近的索引点
3. 从索引点对应位置在 `.log` 文件中顺序扫描到目标 Offset

### Q2：Kafka 为什么不删除已消费的消息？

Kafka 采用日志不可变（immutable log）设计，消息追加后不删除，而是通过 Offset 标记消费进度。这样设计的好处：
- 支持消息回放和重复消费
- 简化存储实现，只需顺序追加
- 通过 Segment 文件整体删除实现清理，效率高
