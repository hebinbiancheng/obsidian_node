# Kafka 消费者与消费组

## 一句话结论

Kafka 消费者通过消费者组实现负载均衡与容错，使用 Pull 模式拉取消息，Offset 管理决定消费进度，重平衡机制在组成员变化时重新分配分区，需注意重复消费与顺序性问题。

## 消费者组

### 核心概念

消费者组是 Kafka 实现**单播**和**广播**两种消息模型的手段：

| 模型 | 说明 |
|------|------|
| **组内单播** | 同一消费者组内，一个 Partition 只能被一个 Consumer 消费 |
| **组间广播** | 不同消费者组独立消费同一 Topic 的全部消息 |

> 消费者组 = 可扩展 + 容错的消费者机制。组内实例挂掉后，其他实例自动接管其分区。

### 关键约束

- 消费者数量 ≤ 分区数量（多余的消费者会空闲）
- 同一组内每个消费者被分配不同的分区
- 不同组的消费者互不影响

## Pull vs Push 模式

Kafka 消费者采用 **Pull 模式**：

| 模式 | 说明 | 优缺点 |
|------|------|--------|
| **Push**（生产者→Broker） | Producer 将消息推送到 Broker | — |
| **Pull**（Broker→Consumer） | Consumer 从 Broker 拉取消息 | 消费者按自己能力消费，避免被压垮 |

**Pull 模式缺点**：Broker 无消息时消费者空轮询。解决方案：Kafka 提供参数让消费者阻塞等待新消息到达。

**为什么不用 Push？** Push 模式下 Broker 决定推送速率，当推送速率远超消费速率时，消费者会被压垮。

## Offset 管理

| 概念 | 说明 |
|------|------|
| **Offset** | 每条消息在 Partition 中的唯一 ID，单调递增不可变 |
| **提交** | Consumer 将已消费的 Offset 上报给 Kafka |

### 自动提交 vs 手动提交

| 方式 | 配置 | 优点 | 缺点 |
|------|------|------|------|
| **自动提交** | `enable.auto.commit=true` | 简单，无需编码 | 可能重复消费或丢失消息 |
| **手动提交** | `enable.auto.commit=false` | 精确控制，可靠性高 | 需手动编码 |

> **推荐**：生产环境使用手动提交，确保消息处理成功后再提交 Offset。

### Offset 读取流程

1. 连接 ZooKeeper，获取 Topic 的 Partition 信息和 Leader 位置
2. 连接到 Leader 对应的 Broker
3. Consumer 将保存的 Offset 发送给 Leader
4. Leader 根据 Offset 定位到 Segment（索引文件 + 日志文件）
5. 通过索引定位日志文件中对应位置，读取数据返回

## 重平衡机制

### 触发条件

| 条件 | 说明 |
|------|------|
| 消费者组成员数量变化 | 新增或移除消费者 |
| 订阅主题数量变化 | 使用正则订阅时新增匹配 Topic |
| 订阅主题分区数变化 | 动态增加分区 |

### 重平衡流程

1. **暂停消费**：所有消费者停止拉取操作
2. **计算分配方案**：根据消费者数量和分区数计算分配
3. **通知消费者**：将分配方案发送给每个消费者
4. **重新分配分区**：应用分区分配方案
5. **恢复消费**：消费者开始消费分配到的分区

> ⚠️ **STW 问题**：重平衡期间所有消费者停止消费，目前无好的解决方案，只能尽量避免触发。

### 消费者状态流转

```
Empty → PreparingRebalance → CompletingRebalance → Stable
  ↑                                                    │
  └────────────────────────────────────────────────────┘
                        Dead
```

| 状态 | 说明 |
|------|------|
| Empty | 组内无成员，但可能存在未过期的位移数据 |
| Dead | 组元数据已被协调者移除 |
| PreparingRebalance | 准备重平衡，所有成员需重新加入 |
| CompletingRebalance | 成员已加入，等待分配方案 |
| Stable | 重平衡完成，正常消费 |

## 分区分配策略

| 策略 | 配置值 | 原理 |
|------|--------|------|
| **Range** | `RangeAssignor`（默认） | 按主题分区序号排序，均分，前几个消费者多分一个 |
| **RoundRobin** | `RoundRobinAssignor` | 所有分区按字典序排序，轮询分配 |
| **Sticky** | `StickyAssignor` | 保留之前分配结果，尽量减少分区移动 |

## 顺序消费

### Kafka 内部顺序保证

- **同一 Partition 内消息有序**：消息追加到日志文件末尾，按 Offset 顺序消费
- **跨 Partition 无序**：不同 Partition 之间无顺序保证

### 实现方案

| 方案 | 做法 | 适用场景 |
|------|------|---------|
| **全局有序** | Topic 只设 1 个 Partition | 严格要求全局顺序 |
| **局部有序** | 指定 Partition Key，相同 Key 进入同一 Partition | 同一业务实体有序 |
| **多线程有序** | 相同 Key 的消息放入同一内存队列，单线程消费 | 提高吞吐同时保证顺序 |

## 重复消费

### 重复消费场景

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| 消费者故障 | 手动提交前崩溃 | 幂等性 / EOS |
| 自动提交间隔内失败 | 已处理但未提交 | 手动控制提交 |
| 生产者重试 | 未启用幂等性 | 启用幂等性 |
| 消费者组再平衡 | 偏移量管理不当 | 消费者幂等处理 |

### 幂等性方案

**一锁、二判、三更新**：

| 步骤 | 做法 |
|------|------|
| 先加锁 | Redis 分布式锁（高性能），可引入锁分段 |
| 幂等性判断 | 基于状态机、流水表、唯一性索引检查 |
| 数据更新 | 持久化后释放锁 |

**其他方案**：
- 数据库唯一键约束
- Redis Set 天然幂等
- 消息携带全局唯一 ID，消费前查 Redis 判断

## 常见问题

### Q1：Kafka 消费过的消息如何再消费？

Kafka 消费的 Offset 定义在 ZooKeeper（或内部 Topic）中。如需重复消费，可在 Redis 中自行记录 checkpoint 点，通过读取 checkpoint 重设 Offset 实现重复消费。

### Q2：消费者组重平衡有什么影响？如何避免？

重平衡期间所有消费者停止消费（STW），影响实时性。避免方法：减少消费者组变化频率、合理设置 `session.timeout.ms` 和 `max.poll.interval.ms`、避免处理时间过长导致心跳超时。
