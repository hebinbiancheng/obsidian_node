# Kafka 核心概念与架构

## 一句话结论

Kafka 是分布式流处理平台，由 Producer、Broker 集群、Consumer 三大组件构成，通过 Topic/Partition 实现消息的逻辑分类与物理分片，依赖 ZooKeeper（或 KRaft）进行集群协调。

## Kafka 是什么

Apache Kafka 是一款**分布式流处理平台**，用于实时构建流处理应用。它本质上是一个**分布式的、可分区的、多副本的、基于发布-订阅模式的消息系统**。

### 核心特点

| 特点 | 说明 |
|------|------|
| **高吞吐量** | 支持每秒百万级消息写入，适用于实时数据管道 |
| **高可用性** | 通过分区和复制机制保证数据冗余和容错 |
| **持久化存储** | 使用磁盘存储消息，支持持久化和回放 |
| **水平扩展** | 通过增加 Broker 实现线性扩展 |

### 常见使用场景

| 场景 | 说明 |
|------|------|
| 实时数据流 | 实时日志收集和分析、监控数据流处理 |
| 事件驱动架构 | 系统间的事件通知和消息传递 |
| 数据集成 | 不同数据源间的数据同步和集成 |
| 日志和指标收集 | 集中收集应用程序日志和指标 |

## 核心组件

### 整体架构

Kafka 的整体架构是**显式分布式架构**，主要由 Producer、Broker 集群和 Consumer 组成。

| 组件 | 角色 | 说明 |
|------|------|------|
| **Producer** | 消息生产者 | 将消息发布到 Kafka 集群中的一个或多个 Topic |
| **Broker** | Kafka 服务器 | 集群中的每个节点，存储 Topic 的分区副本 |
| **Consumer** | 消息消费者 | 从 Kafka 集群订阅并消费消息 |
| **ZooKeeper** | 协调服务 | 维护集群元数据、Controller 选举、成员管理 |

### Topic 与 Partition

| 概念 | 层级 | 说明 |
|------|------|------|
| **Topic** | 逻辑层 | 消息的逻辑容器，用于区分业务。如订单 Topic、日志 Topic |
| **Partition** | 物理层 | Topic 的物理分区，每个 Partition 是一个有序且持久化的日志文件 |

**为什么有了 Topic 还需要 Partition？**

| 好处 | 说明 |
|------|------|
| **提升吞吐量** | 多个 Partition 可并行处理，提高系统吞吐 |
| **负载均衡** | 消费者组内各消费者均匀分配 Partition |
| **扩展性** | 增加 Partition 数量即可水平扩展 |

### Broker 与副本

| 角色 | 说明 |
|------|------|
| **Leader Broker** | 分区的主副本，负责处理消息读写请求 |
| **Follower Broker** | Leader 的备份节点，被动同步数据，Leader 故障时参与选举 |

> 每个分区只有一个 Leader 和多个 Follower，只有 Leader 处理读写请求。Leader 宕机后，Kafka 自动从 ISR 中选举新 Leader。

### ZooKeeper 的作用

| 职责 | 说明 |
|------|------|
| 存放元数据 | 主题分区的所有数据以 ZooKeeper 为权威 |
| 成员管理 | Broker 节点的注册、注销及属性变更 |
| Controller 选举 | 选举集群 Controller |
| 其他管理 | 主题删除、参数配置等 |

> **趋势**：KIP-500 提案完成后，Kafka 将使用基于 Raft 的 KRaft 共识算法替代 ZooKeeper。

## 为什么要使用消息队列（Kafka）

| 作用 | 说明 |
|------|------|
| **缓冲和削峰** | 上游突发流量时，Kafka 暂存消息，下游按自己节奏处理 |
| **解耦和扩展性** | 消息队列作为接口层，解耦业务流程，便于扩展 |
| **冗余** | 一对多发布-订阅，多个业务可独立消费同一消息 |
| **健壮性** | 消息可堆积，消费端短时故障不影响主业务 |
| **异步通信** | 允许消息放入队列后异步处理，提升响应速度 |

## Kafka vs 传统消息系统

| 维度 | Kafka | 传统 MQ |
|------|-------|---------|
| 消息保留 | 持久化日志，可重复读取，无限期保留 | 消费后删除 |
| 架构 | 分布式，灵活伸缩 | 多为单机或简单集群 |
| 流处理 | 原生支持实时流式处理 | 通常不支持 |

## 常见问题

### Q1：Kafka 的整体架构是怎样的？

Kafka 由 Producer、Broker 集群、Consumer 三部分组成，依赖 ZooKeeper 协调。Producer 将消息发送到指定 Topic 的某个 Partition，Broker 负责存储和复制，Consumer 从 Broker 拉取消息消费。每个 Partition 有 Leader 和多个 Follower 副本，Leader 处理读写，Follower 同步备份。

### Q2：Kafka 和 Flume 如何选择？

- **Kafka**：通用分布式消息队列，适合数据被多个系统消费的场景
- **Flume**：专用工具，针对 HDFS/HBase 优化，集成 Hadoop 安全特性
- 建议：数据被多个系统消费用 Kafka，数据只给 Hadoop 用 Flume
