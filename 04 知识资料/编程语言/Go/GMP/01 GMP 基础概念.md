# GMP 基础概念

## 1. 线程、协程、goroutine

### 线程 Thread

线程是操作系统内核视角下的最小调度单元。线程的创建、销毁、切换和调度都需要内核参与。

特点：

- 调度由操作系统完成；
- 切换成本较高；
- 与 CPU 调度模型天然契合；
- 数量过多时会带来较大资源开销。

### 协程 Coroutine

协程可以理解为用户态线程，是用户程序对线程概念的二次封装。协程调度通常在用户态完成，不需要每次都进入内核。

特点：

- 更轻量；
- 切换成本低；
- 调度由用户态运行时负责；
- 需要语言或运行时提供额外调度机制。

### goroutine

goroutine 是 Go 对协程的工程化实现，但它不能孤立理解。goroutine 的能力依附于 GMP 调度体系。

相比普通协程，goroutine 的优势包括：

- G、M、P 可以动态组合，调度灵活；
- goroutine 栈可以动态扩缩；
- Go 将并发粒度统一到 G，屏蔽了线程细节；
- 网络 IO、阻塞、抢占、GC 等都围绕 GMP 体系协同。

## 2. GMP 三元组

`GMP = Goroutine + Machine + Processor`。

| 组件 | 含义 | 类比 | 作用 |
|---|---|---|---|
| G | goroutine | 任务 | 承载用户函数、栈、状态 |
| M | machine / OS thread | 引擎 | 真正执行代码的线程 |
| P | processor | 调度中枢 / 令牌 | 持有本地队列，决定并行调度能力 |

## 3. G：任务

G 是 Go 中对 goroutine 的抽象。

它包含：

- 运行栈；
- 生命周期状态；
- 待执行函数；
- 调度上下文；
- 与 M/P 的关联信息。

理解方式：G 是被调度系统分配和执行的“任务”。

## 4. M：执行引擎

M 是 Go runtime 对操作系统线程的抽象。

M 的运行对象在两类 G 之间切换：

- `g0`：执行调度逻辑；
- 普通 `g`：执行用户代码。

M 本身不直接决定调度秩序，M 需要绑定 P 后才能进入 GMP 调度体系。

## 5. P：调度资源

P 是 Go runtime 的调度器资源，也是本地运行队列的持有者。

关键点：

- P 的数量决定最大并行执行 G 的数量；
- P 的数量受 `GOMAXPROCS` 控制；
- M 必须绑定 P 才能执行普通 G；
- P 持有本地运行队列 `lrq`。

## 6. 运行队列

G 的就绪队列分为两类：

| 队列 | 名称 | 特点 |
|---|---|---|
| lrq | local run queue | 每个 P 私有，优先使用，竞争少，通常无锁/CAS |
| grq | global run queue | 全局共享，竞争更激烈，访问需要全局锁 |

### 放入 G：就近原则

当某个 G 通过 `go func(){}` 创建子 G 时：

1. 优先放入当前 P 的本地队列；
2. 如果本地队列满了，再放入全局队列。

### 获取 G：负载均衡

调度时，g0 查找可运行 G 的大致顺序：

1. 当前 P 的本地队列；
2. 全局队列；
3. netpoll 中 IO 就绪的 G；
4. 从其他 P 的本地队列偷取。

补充细节：每经过一定次数调度后，会优先检查一次全局队列，避免全局队列饥饿。

## 7. GMP 生态

GMP 不只是调度模型，还影响 Go runtime 的多个模块：

- 内存管理：每个 P 有自己的 `mcache`，减少内存分配竞争；
- GC：STW、协助标记、写屏障等与 G/P/M 状态有关；
- netpoll：把网络 IO 阻塞控制在 G 粒度，而不是线程粒度；
- channel / mutex：阻塞时通过 `gopark` 让渡，唤醒时通过 `goready` 回到就绪队列。

## 记忆口诀

```text
G 是任务，M 是线程，P 是调度令牌。
M 要跑 G，必须先拿 P。
G 优先进本地队列，找不到再去全局、netpoll、偷别人。
g0 负责找活，普通 g 负责干活。
```
