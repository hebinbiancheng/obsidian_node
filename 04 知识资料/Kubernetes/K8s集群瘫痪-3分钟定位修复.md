# K8s 集群突然瘫痪！3分钟定位根因 + 完整修复方案

> 原文: [微信文章](https://mp.weixin.qq.com/s/k2y6dTwfkQLSla7GxDT8yw)

---

一张图讲透k8s集群突然瘫痪！3分钟定位根因 + 完整修复方案
K8s 集群整体瘫痪（
kubectl 命令卡死、所有节点 NotReady、Pod 全停、无法调度
），
90% 根因集中在 4 类：etcd 磁盘满/异常、控制平面组件崩溃、节点网络/资源耗尽、CNI 网络插件全崩
，下面这份3分钟分层定位流程图+命令+修复方案，直接收藏，故障时照做就能快速救回集群，不用再到处查资料，流程图也提供给大家，可以进行参考
[IMG]
一、3分钟快速定位（按顺序执行，一步都别乱）
第1分钟：判断集群整体状态（先看控制平面是否存活）——现象：kubectl get nodes 超时/无响应、kubectl cluster-info 失败，核心命令直接复制执行：
bash
kubectl get nodes -o wide
kubectl get pods -n kube-system -o wide
kubectl get events -A --sort-by='.lastTimestamp' | tail -20
说明：依次为检查节点状态、检查kube-system核心Pod（apiserver/etcd/controller-manager/scheduler）、查看集群关键事件。
结论一眼分清：所有命令卡死→控制平面（apiserver/etcd）全挂（最致命）；节点NotReady，但命令可用→节点层/网络/资源问题；部分Pod异常，节点Ready→非整体瘫痪，属应用/调度问题，不用慌。
第2分钟：控制平面根因定位（集群瘫痪最高发，直接登录Master节点执行）
1. etcd故障（占瘫痪60%+），命令速查：
bash
systemctl status etcd
journalctl -u etcd -f --no-pager | grep -E "panic|error|space|corrupt|timeout"
df -h /var/lib/etcd
ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
--cacert=/etc/kubernetes/pki/etcd/ca.crt \
--cert=/etc/kubernetes/pki/etcd/server.crt \
--key=/etc/kubernetes/pki/etcd/server.key endpoint status
典型错误对应根因：
bash
mvcc: database space exceeded / no space → etcd磁盘100%
etcdserver: leader lost / connection refused → etcd集群分裂/崩溃
corrupted database → etcd数据损坏
2. kube-apiserver故障，命令速查：
bash
systemctl status kube-apiserver
journalctl -u kube-apiserver -f | grep -E "fatal|error|connection refused"
常见原因：无法连接etcd（证书/地址错）、6443端口被占用、配置文件语法错误（/etc/kubernetes/manifests/kube-apiserver.yaml）。
3. controller-manager / scheduler故障，命令速查：
bash
systemctl status kube-controller-manager kube-scheduler
journalctl -u kube-controller-manager -f
影响：Pod无法调度、副本不恢复、节点状态不更新，及时重启即可缓解。
第3分钟：节点层/网络根因定位（命令可用但节点NotReady，在Master/Worker节点执行）
1. kubelet异常，命令速查：
bash
systemctl status kubelet
journalctl -u kubelet -f | grep -E "error|panic|failed"
top # 查看CPU/内存占用
free -m # 查看内存剩余
df -h # 查看磁盘占用
常见原因：节点CPU/内存/磁盘100%、CNI插件崩溃、Docker/containerd挂掉、证书过期。
2. 网络全崩（Pod无IP、跨节点不通），命令速查：
bash
kubectl get pods -n kube-system | grep -E "calico|flannel" # 检查CNI Pod状态
ip addr # 查看节点网络
route -n # 查看路由
iptables -L -n # 查看防火墙规则
systemctl status firewalld # 检查防火墙状态
二、最常见4大瘫痪根因 + 完整修复方案（照做必恢复）
1. etcd磁盘满（最常见，生产高发）——现象：kubectl全卡死、etcd报no space
修复步骤（按顺序执行）：
bash
1. 紧急扩容/清理磁盘（优先保证etcd运行）
云环境：在线扩容etcd所在磁盘
物理机：删除/var/lib/etcd外无用文件（不删etcd数据）
df -h /var/lib/etcd # 再次确认磁盘空间
2. 手动压缩etcd数据（必做，释放空间）
rev=$(ETCDCTL_API=3 etcdctl endpoint status --write-out=json | jq -r '.[].revision')
ETCDCTL_API=3 etcdctl compact $rev
ETCDCTL_API=3 etcdctl defrag
3. 配置自动压缩（永久避免）
修改etcd启动参数，添加--auto-compaction-retention=24（保留24h数据）
重启etcd
systemctl restart etcd
4. 验证etcd状态
ETCDCTL_API=3 etcdctl endpoint status
2. 控制平面组件崩溃（apiserver/etcd/controller-manager/scheduler）
修复步骤：
bash
1. 检查配置文件（无语法错误）
cat /etc/kubernetes/manifests/kube-apiserver.yaml
cat /etc/kubernetes/manifests/etcd.yaml
2. 重启核心组件
systemctl restart etcd kube-apiserver kube-controller-manager kube-scheduler
3. 高可用集群：逐个重启Master节点（不同时停所有etcd）
4. 数据损坏：从最近快照恢复etcd（提前备份必做！）
ETCDCTL_API=3 etcdctl snapshot restore 快照文件路径 --data-dir=/var/lib/etcd-backup
systemctl restart etcd
3. 节点资源耗尽（CPU/内存/磁盘100%）
修复步骤：
bash
1. 登录异常节点，杀掉高占用进程
top # 查看进程，记住PID
kill -9 高占用PID（非系统核心进程）
2. 临时驱逐恶意Pod（无资源限制的Pod）
kubectl delete pod
-n
--grace-period=0 --force
3. 清理磁盘/镜像
docker system prune -a # 清理无用镜像
rm -rf /var/log/pods/* # 清理Pod日志（谨慎，避免误删有用日志）
4. 配置资源限制（永久避免）
命名空间添加ResourceQuota、LimitRange，限制Pod资源占用
4. CNI网络插件全崩（Calico/Flannel）——现象：Pod无IP、跨节点Pod不通、节点NotReady
修复步骤（以Calico为例，Flannel操作类似）：
bash
1. 删除异常CNI Pod，重建Calico
kubectl delete -f https://docs.projectcalico.org/manifests/calico.yaml
kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml
2. 检查网络配置（无冲突）
ip addr # 确认容器网段（10.244.0.0/16）与节点网段不重叠
calicoctl node status # 检查BGP网络正常
3. 关闭防火墙/SELinux（避免拦截）
systemctl stop firewalld
setenforce 0
4. 重启kubelet，恢复节点网络
systemctl restart kubelet
三、集群恢复验证（修复后必做，避免假恢复）
bash
1. 验证节点状态（全Ready）
kubectl get nodes
2. 验证核心组件（全Running）
kubectl get pods -n kube-system
3. 验证Pod调度（正常创建）
kubectl run test-nginx --image=nginx --restart=Never
kubectl get pods test-nginx
4. 验证网络连通性
kubectl exec test-nginx -- ping baidu.com
5. 清理测试Pod
kubectl delete pod test-nginx
四、预防机制（避免再次瘫痪）
1. etcd高可用：3个etcd节点+定期备份（每天1次快照），配置自动压缩；
2. 资源监控：配置CPU/内存/磁盘/etcd空间阈值告警（阈值80%）；
3. 强制资源限制：所有命名空间配置ResourceQuota、LimitRange；
4. 日志管理：配置logrotate日志轮转，禁止无限日志占用磁盘；
5. 高可用控制平面：3个Master节点，避免单点故障；
6. 定期检查：每周检查CNI网络、证书有效期、核心组件状态。

## 速查卡

### 第1分钟：判断整体状态

| 现象 | 根因方向 |
|------|----------|
| `kubectl get nodes` 全部卡死 | 控制平面全挂（apiserver/etcd）|
| 节点 NotReady，命令可用 | 节点层/网络/资源问题 |
| 部分 Pod 异常，节点 Ready | 应用/调度问题，非瘫痪 |

### 第2分钟：控制平面定位

| 组件 | 关键命令 | 高频根因 |
|------|----------|----------|
| etcd（60%+） | `systemctl status etcd` + `df -h /var/lib/etcd` | 磁盘满、集群分裂、数据损坏 |
| apiserver | `journalctl -u kube-apiserver` | etcd 连不上、端口占用、YAML 语法错 |
| controller/scheduler | `systemctl status` + `journalctl` | 重启即可恢复 |

### 第3分钟：节点/网络定位

| 组件 | 关键命令 | 高频根因 |
|------|----------|----------|
| kubelet | `journalctl -u kubelet` + `top/df` | 资源耗尽、CNI 挂、containerd 挂、证书过期 |
| CNI 网络 | `kubectl get pods -n kube-system \| grep calico/flannel` | 插件崩溃、网段冲突、防火墙拦截 |

### 四大根因修复速查

| 根因 | 现象 | 修复核心 |
|------|------|----------|
| 1. etcd 磁盘满 | kubectl 全卡死 | 清理磁盘 → `etcdctl compact + defrag` → 配置自动压缩 |
| 2. 控制平面崩溃 | 组件挂掉 | 检查 YAML → `systemctl restart` → 快照恢复 |
| 3. 节点资源耗尽 | 节点 NotReady | kill 高占用进程 → 驱逐 Pod → 配置 ResourceQuota |
| 4. CNI 插件全崩 | Pod 无 IP | `kubectl delete/apply` CNI → 重启 kubelet |

### 恢复验证

```bash
kubectl get nodes                    # 全都 Ready 才行
kubectl get pods -n kube-system      # 核心组件全 Running
kubectl run test-nginx --image=nginx # 能正常调度
kubectl exec test-nginx -- ping baidu.com  # 网络通
```

### 预防机制

- etcd 3节点 + 每日快照 + 自动压缩
- 资源监控告警（阈值 80%）
- 所有命名空间强制 ResourceQuota + LimitRange
- logrotate 日志轮转
- 3 Master 高可用
- 每周检查 CNI、证书、核心组件
