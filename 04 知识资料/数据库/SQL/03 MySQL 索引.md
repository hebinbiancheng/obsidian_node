# 03 MySQL 索引

## 一句话结论

MySQL（InnoDB/MyISAM）使用 B+Tree 作为默认索引结构，兼顾范围查询和磁盘 IO 效率。理解聚簇/非聚簇索引、覆盖索引、最左前缀匹配和索引下推，是掌握索引优化的关键。

---

## 索引是什么

索引是一种**排序好的数据结构**，用于快速查询和检索数据，本质相当于书的目录。

- 优点：大幅加快检索速度，减少 IO；唯一索引保证数据唯一性
- 缺点：创建和维护索引耗时；增删改时索引需动态修改；数据量小时提升不明显

---

## 索引底层数据结构演进

| 结构 | 特点 | 为何不选 |
|------|------|----------|
| **Hash 表** | O(1) 等值查询 | ❌ 不支持范围/排序查询 |
| **BST（二叉查找树）** | 平衡时 O(logN) | ❌ 不平衡时退化为 O(N) 链表 |
| **AVL 树** | 严格平衡 O(logN) | ❌ 频繁旋转开销大，每节点一次 IO |
| **红黑树** | 大致平衡，插入快 | ❌ 树高度较高，磁盘 IO 次数多 |
| **B 树** | 多路平衡，节点存 key+data | ❌ 范围查询需中序遍历 |
| **B+Tree** ✅ | 数据只在叶子节点，叶子链表 | ✅ IO 少、查询稳定、范围查询快 |

### B 树 vs B+Tree 对比

| 维度 | B 树 | B+Tree |
|------|------|--------|
| 数据存储 | 所有节点存 key + data | 仅叶子节点存 key + data |
| 叶子节点 | 独立 | 有引用链连接相邻叶子 |
| 查询效率 | 不稳定（可能中途结束） | 稳定（始终到叶子） |
| 范围查询 | 中序遍历 | 遍历链表即可 |
| IO 次数 | 较多 | 更少 |

## 索引类型总结

### 按数据结构

| 类型 | 说明 |
|------|------|
| BTree 索引 | 默认和最常用，B+Tree 实现 |
| Hash 索引 | 等值查询快，不支持范围（InnoDB 自适应 Hash） |
| 全文索引 | 文本分词搜索，一般用 ES 替代 |
| RTree 索引 | 空间数据，一般用 ES 替代 |

### 按存储方式

| 类型 | 说明 | 引擎 |
|------|------|------|
| **聚簇索引** | 索引+数据一起存放，叶子存完整行 | InnoDB 主键索引 |
| **非聚簇索引** | 索引和数据分开，叶子存主键值或地址 | MyISAM 全部索引 / InnoDB 二级索引 |

### 按应用维度

| 类型 | 特点 |
|------|------|
| 主键索引 | 唯一 + 非空 + 每表一个 |
| 唯一索引 | 唯一 + 允许 NULL + 可多个 |
| 普通索引 | 仅加速查询 |
| 覆盖索引 | 索引包含查询所需全部字段，无需回表 |
| 联合索引 | 多列组成，遵循最左前缀 |
| 前缀索引 | 字符串前 N 个字符建索引 |

---

## 聚簇索引 vs 非聚簇索引

| 维度 | 聚簇索引 | 非聚簇索引 |
|------|----------|------------|
| 数据存放 | 叶子节点存完整行数据 | 叶子节点存主键值（或地址） |
| 查询速度 | 极快（一次定位） | 可能需要回表（二次查询） |
| 更新代价 | 大（叶子存数据） | 较小 |
| 数量限制 | 每表仅一个 | 可多个 |
| 代表 | InnoDB 主键索引 | InnoDB 二级索引 / MyISAM 全部索引 |

### 回表
使用非聚簇索引查询时，先通过索引找到主键值，再通过主键索引定位完整行数据的过程。

### 覆盖索引避免回表
当查询的字段全部在索引中时，直接从索引获取数据，无需回表。

---

## 联合索引与最左前缀匹配

### 最左前缀匹配原则

使用联合索引时，MySQL 从索引的最左列开始，**从左到右依次匹配**查询条件，直到遇到范围查询（`>` `<` `BETWEEN` `LIKE`）时停止。

> 例外：`>=` `<=` `BETWEEN` 以及前缀 `LIKE 'abc%'` 不会停止匹配。

### 示例：联合索引 (a, b, c)

| 查询条件 | 是否走索引 | 说明 |
|----------|------------|------|
| `a=1 AND b=2 AND c=3` | ✅ 全匹配 | 三个字段都用上 |
| `a=1 AND c=3` | ⚠️ 部分 | 只用 a，c 被范围截断 |
| `b=1 AND c=3` | ❌ 不走 | 缺少最左列 a |
| `c=1` | ❌ 不走 | 缺少最左列 a |
| `a=1 AND b>2 AND c=3` | ⚠️ a+b | c 被范围查询截断 |

## 索引下推（ICP）

MySQL 5.6+ 特性：存储引擎在索引遍历时，直接过滤 WHERE 条件中索引可覆盖的部分，**减少回表次数**。

```
无 ICP：索引定位 → 回表取全行 → Server 层过滤
有 ICP：索引定位 + 过滤 → 回表取全行（仅符合条件的行）
```

**适用条件**：InnoDB/MyISAM、range/ref/eq_ref 查询、仅非聚簇索引。

## 索引优化建议

| 建议 | 说明 |
|------|------|
| 选合适字段 | 不为 NULL、频繁查询、WHERE 条件、排序、连接字段 |
| 慎重建索引 | 频繁更新的字段不建议建索引 |
| 控制数量 | 单表索引不超过 5 个 |
| 优先联合索引 | 节省磁盘空间，提升修改效率 |
| 避免冗余索引 | `(a,b)` 已覆盖 `(a)` |
| 前缀索引 | 字符串字段用前缀索引替代普通索引 |
| 删除无用索引 | MySQL 5.7 可查 `sys.schema_unused_indexes` |

### 索引失效常见场景

1. 未遵守最左前缀匹配
2. 索引列上进行计算、函数、类型转换
3. `LIKE '%xxx'` 以 `%` 开头
4. OR 前后条件有一个列无索引
5. IN 取值范围过大 → 全表扫描
6. 隐式类型转换
7. `SELECT *` 可能导致无法使用覆盖索引

## 常见问题

### Q1：为什么 MySQL 选择 B+Tree 而不是 Hash 或红黑树？

Hash 不支持范围查询和排序；红黑树高度较高，磁盘 IO 次数多。B+Tree 是多路平衡树，高度低（3-4 层即可存千万数据），叶子节点形成有序链表，支持高效范围查询，且每次查询 IO 次数稳定。

### Q2：InnoDB 为什么建议使用自增主键？

自增主键保证数据按顺序写入，避免 B+Tree 的页分裂。如果用 UUID 等随机值，插入时会导致频繁的页分裂和页内数据移动，严重影响写入性能。同时，自增主键作为聚簇索引，二级索引叶子存的是主键值，主键越小，二级索引占用空间越小。
