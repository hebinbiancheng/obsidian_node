# Redis 缓存设计

## 一句话结论

缓存设计三大核心问题：穿透（查不存在的数据）、击穿（热点过期）、雪崩（大量同时过期），以及缓存与数据库的双写一致性，是面试和实际开发的高频考点。

## 缓存三大问题

### 缓存穿透

**定义**：查询不存在的数据，缓存和数据库都没有，请求直达数据库。

**原因**：
- 业务误操作（缓存和数据库数据都被误删）
- 恶意攻击（大量请求不存在的数据）

**解决方案**：

| 方案 | 说明 |
|------|------|
| **非法请求限制** | API 入口校验参数合法性 |
| **缓存空值/默认值** | 对不存在的数据也缓存（设短过期时间） |
| **布隆过滤器** | 快速判断数据是否存在，不存在直接返回 |

### 缓存击穿

**定义**：热点数据过期瞬间，大量请求直达数据库。

**解决方案**：

| 方案 | 说明 |
|------|------|
| **互斥锁** | SETNX 实现，同一时间只有一个线程重建缓存 |
| **永不过期** | 热点数据不设过期时间，后台异步更新 |
| **提前刷新** | 过期前通知后台线程更新缓存 |

### 缓存雪崩

**定义**：大量缓存同时过期或 Redis 节点故障，请求全部打到数据库。

**场景**：
- 大量数据同时过期
- Redis 节点故障

**解决方案**：

| 方案 | 说明 |
|------|------|
| **过期时间随机化** | 在基础过期时间上加随机值（1-10 分钟） |
| **缓存永不过期** | 后台服务异步更新 |
| **降级熔断** | 非核心数据返回预定义信息/空值 |
| **高可用架构** | 哨兵/集群模式，多实例部署 |

## 缓存更新策略

### Cache Aside（旁路缓存）— 最常用

**读策略**：
```
读缓存 → 命中返回 / 未命中 → 读数据库 → 写入缓存 → 返回
```

**写策略**：
```
先更新数据库 → 再删除缓存
```

**为什么先更新数据库再删除缓存？**

| 顺序 | 并发问题 |
|------|---------|
| 先删缓存再更新数据库 | A 删缓存 → B 读数据库（旧值）→ B 写缓存（旧值）→ A 更新数据库（新值）→ **缓存旧，数据库新** |
| 先更新数据库再删缓存 | A 读数据库（旧值）→ B 更新数据库（新值）→ B 删缓存 → A 写缓存（旧值）→ **概率极低**（缓存写入远快于数据库写入） |

**适用场景**：读多写少

### Read/Write Through（读穿/写穿）

- 应用只与缓存交互，缓存代理数据库操作
- Redis 和 Memcached 不原生支持
- 适用场景：本地缓存

### Write Back（写回）

- 只更新缓存，异步批量写数据库
- ✅ 写性能高
- ❌ 数据可能丢失（掉电）
- 适用场景：CPU 缓存、文件系统 Page Cache

## 双写一致性

### 四种同步策略

| 策略 | 评价 |
|------|------|
| 先更新缓存，再更新数据库 | ❌ 不推荐 |
| 先更新数据库，再更新缓存 | ❌ 不推荐 |
| 先删除缓存，再更新数据库 | ⚠️ 需配合延时双删 |
| **先更新数据库，再删除缓存** | ✅ 推荐 |

### 删除失败的重试机制
```
更新数据库成功 → 删除缓存失败 → 写入消息队列 → 异步重试删除
```

### 延时双删
```
删除缓存 → 更新数据库 → sleep(N ms) → 再次删除缓存
```
解决「先删缓存再更新数据库」的并发问题，N 需考虑主从同步延迟。

### 读写分离架构的额外问题
主从同步延迟期间，从库可能返回旧数据。解决方案：延时双删时考虑主从同步时间。

## 热点数据动态缓存策略

**思路**：通过访问时间排名，只缓存 Top N 热点数据。

**实现**：
1. ZSet 排序队列（如 Top 1000 商品）
2. 定期过滤末尾 200 个，从数据库随机补充 200 个
3. 请求先查队列获取 ID，再查缓存获取详情

## 常见问题

- **Q：缓存穿透和缓存击穿的区别？**
  A：穿透是查**不存在**的数据，击穿是**热点数据过期**。前者用布隆过滤器，后者用互斥锁。

- **Q：为什么推荐删除缓存而不是更新缓存？**
  A：更新缓存消耗大（尤其复杂计算场景），删除后下次查询自动重建，且避免并发更新问题。

- **Q：Cache Aside 适合写多场景吗？**
  A：不适合。写频繁会导致缓存频繁删除，命中率低。可考虑加分布式锁更新缓存或设短过期时间。
