# Redis 集群与高可用

## 一句话结论

Redis 高可用三件套：主从复制（读写分离 + 数据冗余）→ 哨兵模式（自动故障转移）→ Cluster（数据分片 + 水平扩展），层层递进解决不同规模的高可用需求。

## 主从复制

### 架构
一主多从，读写分离：主节点写，从节点只读。

### 同步机制
使用 `psync` 命令，依赖三个参数：

| 参数 | 作用 |
|------|------|
| **复制偏移量** | 主从各自累加命令长度，从节点每秒上报 |
| **积压缓冲区** | 主节点固定长度队列（默认 1MB），保存最近命令 |
| **主节点运行 ID** | 40 位十六进制，重启变化后触发全量复制 |

### 全量复制 vs 部分复制
- **全量复制**：初次连接或运行 ID 变化时
- **部分复制**：网络中断恢复后，从积压缓冲区补数据

### 注意
- 异步复制，无法保证强一致性
- 主节点执行完命令即返回客户端，不等从节点

## 哨兵模式（Sentinel）

### 解决的问题
主节点宕机时自动故障转移，无需人工介入。

### 哨兵节点特征
1. 定期监控数据节点和其他哨兵
2. 将故障转移结果通知应用方
3. 将从节点晋升为主节点
4. 客户端连接哨兵获取主节点信息
5. 故障判断由多个哨兵共同完成（防误判）
6. 哨兵节点集合健壮（个别不可用不影响）
7. 哨兵也是 Redis 节点，不存储数据

### 故障转移流程
```
哨兵发现主节点不可达 → 标记主观下线
→ 与其他哨兵协商 → 多数确认 → 标记客观下线
→ 选举 leader 哨兵 → 执行故障转移
→ 通知应用方
```

## Redis Cluster（切片集群）

### 适用场景
单机内存/并发/流量达到瓶颈时。

### 哈希槽（Hash Slot）
- 共 **16384** 个哈希槽
- 映射公式：`slot = CRC16(key) % 16384`
- 每个节点负责一部分槽

### 分配方式
- **平均分配**：`cluster create` 自动分配
- **手动分配**：`cluster addslots` 指定（必须分配完 16384 个）

### 节点通信：Gossip 协议

| 消息类型 | 作用 |
|---------|------|
| **meet** | 通知新节点加入集群 |
| **ping** | 每秒随机选 5 个节点发送，检测在线 + 交换状态 |
| **pong** | 响应 meet/ping，或广播自身状态更新 |
| **fail** | 广播节点下线通知 |

### 通信成本
- 每个节点每秒随机选 5 个节点发 ping
- 若某节点超时过半（cluster-node-timeout/2），也发 ping
- 每个消息携带：slots 数组（2KB）+ 1/10 集群状态（约 1KB/10 节点）

### Cluster 限制
1. key 批量操作仅支持同 slot 的 key
2. 事务仅支持同节点上的多 key
3. 不能将大键值对象映射到不同节点
4. 仅支持 DB0（不支持多数据库空间）
5. 复制结构仅一层（不支持嵌套树状复制）

## 脑裂问题

### 什么是脑裂？
主节点网络故障与从节点失联，但客户端仍能访问。哨兵选举新主节点后，集群出现两个主节点。

### 数据丢失原因
网络恢复后，旧主降级为从 → 清空本地数据 → 全量同步新主 → 故障期间写入的数据丢失。

### 解决方案
```conf
min-slaves-to-write 1      # 至少 1 个从节点连接
min-slaves-max-lag 10      # 主从延迟不超过 10 秒
```
不满足条件时，主节点拒绝写请求。

## 常见问题

- **Q：主从复制为什么是异步的？**
  A：保证主节点性能，不等从节点确认即返回。代价是无法强一致性。

- **Q：哨兵和 Cluster 的区别？**
  A：哨兵解决高可用（自动故障转移），Cluster 解决水平扩展（数据分片）。两者可结合使用。

- **Q：为什么哈希槽是 16384？**
  A：16384 个槽的心跳消息占用约 2KB，足够均匀分布且心跳包不太大。
