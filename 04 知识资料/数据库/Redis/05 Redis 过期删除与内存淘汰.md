# Redis 过期删除与内存淘汰

## 一句话结论

Redis 使用「惰性删除 + 定期删除」组合策略处理过期键，内存不足时通过 8 种淘汰策略（含 LRU/LFU）释放空间，两者配合保证内存高效利用。

## 过期删除策略

### 过期字典
- 保存所有 key 的过期时间
- 查询 key 时先检查过期字典，与当前时间比对

### 惰性删除
- **时机**：客户端访问 key 时检查
- ✅ CPU 友好，几乎无额外开销
- ❌ 内存不友好，过期 key 不被访问则永不释放

### 定期删除
- **时机**：每秒 10 次过期扫描
- **流程**（贪心策略）：
  1. 从过期字典随机抽取 20 个 key
  2. 检查并删除已过期的 key
  3. 若过期比例 > 25%，重复步骤 1
  4. 单次循环不超过 25ms（防阻塞）

### 从库的过期处理
- 从库**不主动**进行过期扫描
- 依赖主库同步 DEL 命令来删除过期 key
- 过期 key 在从库被访问时仍会返回（被动处理）

## 内存淘汰策略（8 种）

### 不淘汰
| 策略 | 行为 |
|------|------|
| **noeviction**（默认） | 内存满后拒绝写入，返回错误 |

### 在过期键中淘汰
| 策略 | 行为 |
|------|------|
| **volatile-random** | 随机淘汰 |
| **volatile-ttl** | 淘汰更早过期的 |
| **volatile-lru** | 淘汰最久未使用的 |
| **volatile-lfu**（4.0+） | 淘汰最少使用的 |

### 在所有键中淘汰
| 策略 | 行为 |
|------|------|
| **allkeys-random** | 随机淘汰 |
| **allkeys-lru** | 淘汰最久未使用的 |
| **allkeys-lfu**（4.0+） | 淘汰最少使用的 |

## LRU vs LFU

### LRU（Least Recently Used）
- **标准 LRU**：链表实现，最新访问移到表头，淘汰表尾
- **Redis 近似 LRU**：
  - 对象头 `lru` 字段记录最后访问时间戳
  - 随机采样 N 个 key（默认 5），淘汰最旧的
  - ✅ 节省空间，避免链表移动开销
  - ❌ 无法解决缓存污染（一次性大量读取污染缓存）

### LFU（Least Frequently Used）
- **核心思想**：按访问频率淘汰，解决 LRU 缓存污染问题
- **Redis 实现**：
  - `lru` 字段（24 bits）分为两段：
    - 高 16bit：`ldt`（Last Decrement Time，访问时间戳）
    - 低 8bit：`logc`（Logistic Counter，访问频次）
  - 淘汰时先比频次，频次相同再比时间

### 对比总结

| 维度 | LRU | LFU |
|------|-----|-----|
| 淘汰依据 | 最近访问时间 | 访问频率 |
| 缓存污染 | 可能（一次性大量读取） | 较好 |
| 实现复杂度 | 低 | 中 |
| 版本 | 所有版本 | Redis 4.0+ |

## 过期时间设计建议

- **热点数据**：不设过期时间，避免缓存击穿
- **批量过期**：附加随机数，避免大量 key 同时过期导致雪崩

## 常见问题

- **Q：定期删除为什么用随机抽样？**
  A：避免遍历所有 key 导致长时间阻塞。贪心策略在过期比例高时自动增加扫描。

- **Q：如何选择内存淘汰策略？**
  A：缓存场景推荐 `allkeys-lru` 或 `allkeys-lfu`；需要精确控制过期用 `volatile-*` 系列。

- **Q：LRU 和 LFU 怎么选？**
  A：访问模式有明显冷热区分选 LFU，一般场景 LRU 足够。
