# 05 运维项目经验——Kubernetes 与容器

> 整理自 [[07 已归档/原始笔记/50个运维项目经验简历素材库|50个运维项目经验简历素材库]]，原文作者：数安智信。

## 一句话结论

Kubernetes 和容器化是云原生时代的核心技能，涵盖微服务容器化改造、K8s 集群高可用、CI/CD 流水线和 DevOps 效能提升，是 DevOps 和云原生岗位最重要的加分项。

---

## 一、容器化与 Kubernetes 项目（5个）

| 项目 | 核心场景 | 技术栈 / 方法 | 可量化成果 |
|------|----------|---------------|------------|
| 微服务容器化改造 | 单体应用部署慢、依赖复杂 | Docker、多阶段构建、镜像扫描、微服务拆分 | 镜像体积下降、启动时间缩短、交付周期缩短 |
| Kubernetes 集群高可用部署 | 需要高可用容器平台 | kubeadm、etcd、Calico、Metrics Server、负载均衡 | 集群可用性提升，API Server 和 etcd 高可用 |
| CI/CD 流水线容器化部署 | 代码提交到部署链路依赖人工 | GitLab CI、Jenkins、Harbor、SonarQube、K8s | 部署频率提升、部署成功率提升 |
| Kubernetes 资源优化 | Pod 资源浪费或频繁驱逐 | VPA、HPA、Prometheus、资源 requests/limits | 资源利用率提升，驱逐事件减少 |
| 有状态应用容器化 | 数据库等有状态服务容器化困难 | StatefulSet、PV/PVC、MySQL 主从、备份策略 | RTO 缩短，数据持久化和故障恢复能力增强 |

### K8s 项目简历模板

```text
针对【单体应用部署慢/K8s 集群单点/资源浪费严重】的问题，
负责【微服务容器化改造/K8s 高可用部署/资源优化】，
基于【Docker 多阶段构建/kubeadm+Calico/VPA+HPA】完成【关键动作】，
最终实现【镜像体积缩小XX%/集群可用性提升至99.9X%/资源利用率提升XX%】。
```

### 容器化改造关键步骤

| 步骤 | 动作 | 关键点 |
|------|------|--------|
| 1. 编写 Dockerfile | 多阶段构建、最小基础镜像 | 安全扫描（Trivy）、镜像瘦身 |
| 2. 配置管理分离 | 环境变量/ConfigMap/Secret | 不要硬编码到镜像 |
| 3. 健康检查 | livenessProbe + readinessProbe | 确保 K8s 能正确管理 Pod 生命周期 |
| 4. 资源限制 | requests + limits | 避免资源争抢和 OOMKilled |
| 5. 日志与监控 | stdout/stderr + Prometheus Exporter | 可观测性从 Day 1 开始 |
| 6. CI/CD 集成 | 自动构建 → 推送镜像 → 部署 | GitOps（ArgoCD）推荐 |

## 二、DevOps 与效能提升项目（5个）

| 项目 | 核心场景 | 技术栈 / 方法 | 可量化成果 |
|------|----------|---------------|------------|
| DevOps 文化转型 | 开发、测试、运维协作割裂 | 每日站会、故障复盘、协作平台、共享责任模型 | 沟通成本下降，交付周期缩短 |
| 技术债务清理 | 脚本无人维护、配置硬编码 | 脚本重构、配置标准化、代码评审、配置审核 | 技术债务减少，新功能交付速度提升 |
| 开发测试环境标准化 | 环境不一致导致缺陷难复现 | Docker、Vagrant、Environment as Code | 环境准备时间缩短，缺陷复现率提升 |
| 运维知识体系建设 | 经验散落个人，交接困难 | Wiki、故障案例库、SOP、知识审核机制 | 新人上手时间缩短，问题解决率提升 |
| AI 运维试点 | 告警多、异常发现滞后 | 机器学习、异常检测、告警降噪、容量预测 | 告警准确率提升，无效告警减少 |

### DevOps 项目简历模板

```text
针对【协作割裂/技术债务积累/环境不一致】的问题，
负责【DevOps 转型推动/技术债务清理/环境标准化】，
基于【协作平台+复盘机制/脚本重构+代码评审/Docker+Environment as Code】完成【关键动作】，
最终实现【交付周期缩短XX%/环境准备从X小时降到X分钟/新人上手时间缩短XX%】。
```

| 阶段 | 特征 | 关键实践 |
|------|------|----------|

| 可重复级 | 部分自动化、有文档 | 脚本化、Wiki |

## 常见问题

### Q1：没有 K8s 生产经验，怎么写容器化项目？

**A：** 可以从以下角度切入：① 用 Docker 做过应用容器化（即使没上 K8s）；② 用 Docker Compose 管理多容器应用；③ 在 Minikube/Kind 上做过学习和实验。关键是体现你理解容器化的价值（环境一致性、快速部署、资源隔离），而不是堆砌 K8s 术语。

### Q2：DevOps 转型项目太"虚"，怎么量化？

**A：** 量化维度：① 交付频率（从月级到周级/日级）；② 部署成功率（从 XX% 到 XX%）；③ 故障恢复时间（MTTR）；④ 新人上手时间；⑤ 跨团队沟通成本（会议时间减少）。即使是"文化"类项目，也要找到可衡量的代理指标。
