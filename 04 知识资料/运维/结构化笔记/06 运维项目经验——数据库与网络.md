# 06 运维项目经验——数据库与网络

> 整理自 [[07 已归档/原始笔记/50个运维项目经验简历素材库|50个运维项目经验简历素材库]]，原文作者：数安智信。

## 一句话结论

数据库和网络是运维的两大核心专业方向——数据库项目体现对关键数据资产的保护和性能调优能力，网络项目体现对链路、协议、流量和连通性的掌控能力。

---

## 一、数据库运维项目（5个）

| 项目 | 核心场景 | 技术栈 / 方法 | 可量化成果 |
|------|----------|---------------|------------|
| 数据库性能优化 | SQL 慢、QPS 低、响应慢 | MySQL、slow log、Explain、索引、pt-query-digest | QPS 提升，查询延迟下降 |
| 数据库高可用架构改造 | 单点数据库可用性不足 | MySQL MGR、ProxySQL、读写分离、故障转移 | 可用性提升，故障时间减少 |
| 数据库备份与恢复体系 | 误删数据、恢复能力不足 | 全量 + 增量 + binlog、恢复演练 | 建立 RPO/RTO，成功恢复误删数据 |
| 数据库迁移项目 | Oracle 到 PostgreSQL 等异构迁移 | CDC、数据一致性校验、性能对比、回归测试 | 停机时间可控，迁移后性能提升 |
| 分库分表实施 | 单表数据过亿、查询慢 | ShardingSphere、分表键设计、水平分表 | 查询从秒级降到毫秒级，支持业务增长 |

### 数据库项目简历模板

```text
针对【SQL 慢/单点故障/单表过亿】的问题，
负责【性能优化/高可用改造/分库分表】，
基于【pt-query-digest+Explain/MySQL MGR+ProxySQL/ShardingSphere】完成【关键动作】，
最终实现【QPS 提升XX%/可用性提升至99.9X%/查询从X秒降到X毫秒】。
```

### MySQL 性能优化检查清单

| 优化维度 | 检查项 | 工具/命令 |
|----------|--------|-----------|
| 慢查询 | 开启 slow_query_log，分析慢 SQL | `pt-query-digest`、`mysqldumpslow` |
| 索引 | 检查缺失索引、冗余索引、索引选择性 | `EXPLAIN`、`SHOW INDEX` |
| 配置 | buffer_pool_size、连接数、IO 配置 | `SHOW VARIABLES` |
| 锁 | 检查行锁等待、死锁 | `SHOW ENGINE INNODB STATUS` |
| 硬件 | 磁盘 IOPS、内存、CPU | `iostat`、`vmstat` |

### 数据库高可用方案对比

| 方案 | 原理 | RTO | 复杂度 | 适用场景 |
|------|------|-----|--------|----------|
| 主从 + 手动切换 | 异步复制，人工切换 | 分钟~小时 | 低 | 非核心库 |
| MHA / Orchestrator | 自动故障检测 + 切换 | 30s~分钟 | 中 | MySQL 高可用 |
| MySQL MGR | 组复制，Paxos 协议 | 秒级 | 高 | 强一致性场景 |
| ProxySQL + MGR | 读写分离 + 自动路由 | 秒级 | 高 | 读写分离 + 高可用 |

## 二、网络优化项目（5个）

| 项目 | 核心场景 | 技术栈 / 方法 | 可量化成果 |
|------|----------|---------------|------------|
| 网络延迟优化 | 核心业务网络延迟高 | Wireshark、QoS、BBR、链路聚合、路由优化 | 延迟下降，传输效率提升 |
| 无线网络覆盖优化 | 办公区/车间 WiFi 盲区和干扰 | AP 规划、信道优化、802.11ac、负载均衡 | 连接成功率提升，漫游切换时间下降 |
| DNS 架构优化 | DNS 解析慢、可用性不足 | 主从 DNS、智能解析、缓存、DNSSEC、健康检查 | 解析成功率提升，平均解析时间下降 |
| 负载均衡架构升级 | 并发能力不足、单点风险 | F5、Nginx、四层/七层负载均衡、SSL 卸载 | TPS 提升，支撑峰值流量 |
| SDN 网络转型 | 网络开通慢、变更错误多 | OpenFlow、SDN 控制器、拓扑发现、自动化配置 | 网络开通时间缩短，变更错误率下降 |

### 网络项目简历模板

```text
针对【网络延迟高/DNS 解析慢/负载均衡单点】的问题，
负责【延迟优化/DNS 架构升级/负载均衡改造】，
基于【Wireshark+BBR/主从DNS+智能解析/Nginx+F5】完成【关键动作】，
最终实现【延迟下降XX%/解析时间下降XX%/TPS 提升XX%】。
```

### 四层 vs 七层负载均衡

| 维度 | 四层（L4） | 七层（L7） |
|------|-----------|-----------|
| 工作层级 | TCP/UDP | HTTP/HTTPS |
| 代表产品 | F5、LVS、HAProxy（TCP 模式） | Nginx、HAProxy（HTTP 模式）、Traefik |
| 路由依据 | IP + 端口 | URL、Header、Cookie |
| 性能 | 高（包转发） | 中（需解析协议） |
| 灵活性 | 低 | 高（可按 URL 路由、灰度发布） |
| SSL 卸载 | 不支持 | 支持 |

### 网络排障工具速查

| 工具 | 用途 | 示例命令 |
|------|------|----------|
| ping | 连通性测试 | `ping -c 10 target` |
| traceroute | 路由追踪 | `traceroute target` |

## 常见问题

### Q1：数据库性能优化面试常问什么？

**A：** 高频问题：① 慢查询怎么排查（slow log → EXPLAIN → 索引优化 → 配置调优）；② 索引失效的场景（LIKE '%xx'、函数包裹列、类型隐式转换）；③ MySQL 和 PostgreSQL 的差异；④ 分库分表的方案选择（水平分表 vs 垂直分库，分表键怎么选）。

### Q2：网络项目在运维面试中重要吗？

**A：** 重要但常被忽视。面试中网络问题常见：① TCP 三次握手/四次挥手；② DNS 解析全过程；③ 负载均衡四层和七层的区别；④ 一个 HTTP 请求从浏览器到服务器的完整路径。网络是排查"服务不通"类问题的基本功。
