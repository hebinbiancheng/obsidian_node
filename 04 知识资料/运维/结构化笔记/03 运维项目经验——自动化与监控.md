# 03 运维项目经验——自动化与监控

> 整理自 [[07 已归档/原始笔记/50个运维项目经验简历素材库|50个运维项目经验简历素材库]]，原文作者：数安智信。

## 一句话结论

自动化和监控是运维效率的核心杠杆——自动化减少重复劳动、提升一致性，监控保障系统可观测性和故障快速发现，适合中级运维、DevOps 和 SRE 岗位重点展示。

---

## 一、自动化运维项目（5个）

| 项目 | 核心场景 | 技术栈 / 方法 | 可量化成果 |
|------|----------|---------------|------------|
| Ansible 自动化配置管理 | 批量服务器配置不一致 | Ansible Tower、Playbook、template、service 模块 | 配置一致性提升，人工操作减少 |
| 基础设施即代码 IaC 实践 | 多云资源手工创建、环境不一致 | Terraform、HCL、远程状态、Git PR | 资源交付准确率提升，环境差异减少 |
| 批量任务自动化调度 | 定时任务分散、失败无通知 | Rundeck、Shell、Python、邮件/企业微信通知 | 自动化执行率提升，夜间人工处理减少 |
| 智能运维工具开发 | 巡检、故障处理依赖人工 | CMDB、监控数据、Python、故障知识库、自愈脚本 | 故障恢复时间缩短，团队规模不变支撑更多设备 |
| 文档自动化生成 | 运维文档滞后、拓扑信息不准 | Markdown、Git、GitBook、Python、Jenkins | 文档实时更新，准确率提升 |

### 自动化项目简历模板

```text
针对【服务器配置不一致/资源手工创建/定时任务分散】的问题，
负责【Ansible 配置管理/Terraform IaC/智能运维工具开发】，
基于【Ansible Tower/Terraform HCL/Python + CMDB】完成【关键动作】，
最终实现【配置一致性提升至XX%/交付准确率提升XX%/故障恢复时间缩短XX%】。
```

### Ansible vs Terraform 对比

| 维度 | Ansible | Terraform |
|------|---------|-----------|
| 定位 | 配置管理 + 应用部署 | 基础设施即代码（IaC） |
| 工作方式 | SSH 推送，过程式 | API 声明式，状态管理 |
| 状态管理 | 无状态（幂等） | 有状态（state file） |
| 适用场景 | 软件安装、配置、服务管理 | 云资源创建、网络、虚拟机 |
| 学习曲线 | 低（YAML） | 中（HCL） |
| 典型组合 | Ansible 管配置 + Terraform 管资源 | — |

## 二、监控与告警系统项目（5个）

| 项目 | 核心场景 | 技术栈 / 方法 | 可量化成果 |
|------|----------|---------------|------------|
| 全栈监控平台建设 | 缺少统一监控与告警 | Prometheus、Grafana、Exporter、Alertmanager | 故障检测时间从小时级缩短到分钟级 |
| 分布式追踪系统实施 | 微服务调用链难排查 | Jaeger、OpenTelemetry、ELK 联动 | 定位关键瓶颈，跨服务延迟下降 |
| 日志集中分析平台 | 日志分散、排障慢 | ELK、Logstash、Kibana、日志清洗规则 | 问题排查从小时级缩短到分钟级 |
| 告警降噪与分级治理 | 告警过多、误报多 | 告警分级 P0-P3、聚合、抑制、路由 | 无效告警减少，响应效率提升 |
| 业务指标监控体系 | 只监控机器指标，缺少业务视角 | 自定义 Exporter、业务指标、SLA/SLO | 业务异常更早发现，故障影响面降低 |

### 监控项目简历模板

```text
针对【缺少统一监控/日志分散/告警过多】的问题，
负责【全栈监控平台建设/日志集中分析/告警降噪】，
基于【Prometheus+Grafana/ELK/告警分级 P0-P3】完成【关键动作】，
最终实现【故障检测从X小时缩短到X分钟/无效告警减少XX%/排查效率提升XX%】。
```

### 可观测性三支柱

| 支柱 | 工具 | 解决的问题 |
|------|------|------------|
| **Metrics（指标）** | Prometheus + Grafana | 系统是否正常？CPU/内存/QPS 趋势？ |
| **Logging（日志）** | ELK / Loki | 出了什么问题？错误堆栈是什么？ |
| **Tracing（追踪）** | Jaeger / SkyWalking | 请求经过了哪些服务？哪个环节慢？ |

### 告警分级标准

| 级别 | 定义 | 响应时间 | 通知方式 |
|------|------|----------|----------|
| P0 | 核心业务中断，影响用户 | 5 分钟内 | 电话 + 短信 + IM |
| P1 | 重要功能异常，部分用户受影响 | 15 分钟内 | 短信 + IM |
| P2 | 非核心功能异常 | 1 小时内 | IM |
| P3 | 预警/潜在风险 | 工作时间处理 | 邮件/工单 |

## 常见问题

### Q1：Ansible 和 Terraform 应该先学哪个？

**A：** 取决于你的工作场景。如果日常管理大量服务器（配置、软件安装、服务启停），先学 Ansible（YAML 语法简单，上手快）。如果管理云资源（虚拟机、VPC、负载均衡），先学 Terraform。最佳实践是两者结合：Terraform 创建资源，Ansible 配置资源。

### Q2：监控告警太多怎么办？

**A：** 三步走：① 告警分级（P0-P3），不同级别不同响应策略；② 告警聚合（同一故障只发一条）；③ 告警抑制（核心服务挂了就不发依赖服务的告警）。目标是把告警从"狼来了"变成"真正需要行动的信号"。
