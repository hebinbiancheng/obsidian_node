# 数据存储与算法设计

## 一句话结论

海量数据处理需要巧用数据结构（位图、前缀树、DFA）降低时空复杂度，存储设计需要在可读性、空间效率和查询性能之间做权衡。

## 位图（Bitmap）去重

### 场景：40亿个QQ号，1G内存去重

| 方案 | 内存占用 | 可行性 |
|------|----------|--------|
| 直接存储（unsigned int） | 4×40亿 ≈ 14.9 GB | ❌ 远超1G |
| **位图（Bitmap）** | 40亿 bits ≈ 476 MB | ✅ 可行 |

### 原理

- 每个数字只占 1 bit
- 数字 N 对应第 N 个 bit 位
- 存在 → 置 1，不存在 → 保持 0
- 遍历所有为 1 的位置即为去重结果

### 进阶优化：Roaring Bitmap

将 32 位整数拆分为高 16 位 + 低 16 位：
- 高 16 位相同的数字放在一个 container
- 低 16 位形成 bitmap（bucket）
- 稀疏数据大幅降低内存占用

## IP 地址存储方案

### IPv4 存储对比

| 方案 | 存储空间 | 可读性 | 查询/排序性能 | 适用场景 |
|------|----------|--------|--------------|----------|
| **字符串 VARCHAR(15)** | 7-15 字节 | ✅ 直观 | ❌ 字符串比较慢 | 需要展示的场景 |
| **整数 UNSIGNED INT** | 4 字节 | ❌ 需转换 | ✅ 数值比较快 | ⭐ 性能与空间平衡 |
| **二进制 BINARY(4)** | 4 字节 | ❌ 极差 | ✅ 与网络协议一致 | 底层网络操作 |

### IPv6 存储对比

| 方案 | 存储空间 | 说明 |
|------|----------|------|
| **字符串 VARCHAR(39)** | 最多39字节 | 可读性好，空间大 |
| **二进制 BINARY(16)** | 16字节 | ⭐ 推荐，紧凑高效 |
| **整数（2×BIGINT）** | 16字节 | 灵活但需拆分逻辑 |

### MySQL 函数

- IPv4：`INET_ATON()` 转整数，`INET_NTOA()` 转字符串
- IPv6：`INET6_ATON()` 转二进制，`INET6_NTOA()` 转字符串

## 敏感词过滤

### 方案演进

| 方案 | 原理 | 时间复杂度 | 适用场景 |
|------|------|-----------|----------|
| **字符串匹配** | contains / 正则 | O(N×M) | 小规模文本、少量敏感词 |
| **前缀树（Trie）** | 共享前缀的树形结构 | O(N) | 敏感词有共同前缀、大量文本 |
| **DFA（确定有限自动机）** | 状态机驱动的模式匹配 | O(N) | 复杂模式匹配、正则表达式 |

### 前缀树（Trie）实现步骤

1. **构建**：将敏感词逐个插入，共享公共前缀，末尾节点标记为结束
2. **过滤**：从文本每个字符出发，在前缀树中匹配，到达结束节点则命中

### DFA 实现步骤

1. **构建**：为每个敏感词构建状态转移图，最后一个字符对应状态标记为接受状态
2. **过滤**：从初始状态开始，根据输入字符转移状态，到达接受状态则命中

### 开源推荐

[sensitive-word](https://github.com/houbb/sensitive-word) — 基于 DFA 算法的高性能 Java 敏感词过滤框架。

## Redis 实现排行榜（分数相同按时间排序）

### 问题

ZSet 默认分数相同时按 value（用户ID）排序，不是按时间。

### 方案

将 score 设为浮点数：`score = 分数 + 1 - 时间戳/1e13`

- 整数部分 = 得分
- 小数部分 = 时间先后（时间戳越小，小数部分越大，排名越靠前）

## Redis 实现朋友圈点赞

| 需求 | 实现 |
|------|------|
| 点赞/取消点赞 | ZADD / ZREM |
| 点赞数量 | ZCARD |
| 点赞用户列表 | ZREVRANGEBYSCORE（按时间倒序） |
| 去重 | ZSet 天然去重（member=用户ID） |

数据结构：`KEY=朋友圈ID, ZSet{member=用户ID, score=点赞时间戳}`

| 维度 | Redis | MySQL |
|------|-------|-------|
| 性能 | ✅ 高，响应快 | ❌ 相对慢 |
| 可靠性 | ❌ 异步刷盘可能丢数据 | ✅ 持久化可靠 |
| 查询能力 | ❌ 有限 | ✅ 丰富查询 + 事务 |

## 常见问题

### Q1：位图去重有什么局限性？

位图大小由数据范围决定，不是数据量。如果数据范围很大但数据稀疏（如 1 到 100 亿中只有 100 个数字），位图会浪费大量空间。此时可用哈希 + 多个位图（布隆过滤器）或 Roaring Bitmap。

### Q2：前缀树和 DFA 有什么区别？

前缀树是数据结构，DFA 是计算模型。前缀树通过树结构共享前缀来匹配；DFA 通过状态机驱动，更适合复杂模式匹配（如正则表达式）。在敏感词过滤中，DFA 本质上是将前缀树转化为状态机，两者性能接近，DFA 更通用。
