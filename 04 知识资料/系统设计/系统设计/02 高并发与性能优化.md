# 高并发与性能优化

## 一句话结论

高并发场景的核心思路是"逐层过滤 + 分级应对"：正常流量靠扩容和架构优化，异常流量靠防护，单点瓶颈靠缓存、拆分和异步化解。

## 100倍 QPS 突增应对

### 先分析原因和特点

| 情况 | 原因 | 方案 |
|------|------|------|
| **正常增长** | 业务好转、蹭到热点 | 临时：扩容、提升硬件配置 |
| **正常增长** | 长期业务发展 | 长期：设计高并发系统（缓存/异步/分库分表） |
| **异常流量** | DDoS 攻击 | 防 DDoS 手段（流量清洗、黑名单、限流） |

## 单表数据量大：分库分表的替代方案

> **原则**：不到万不得已，不建议直接做分库分表（实现复杂、跨库事务、分页查询等问题）。

### 优先级排序

| 优先级 | 方案 | 说明 |
|--------|------|------|
| 1 | **数据库优化** | 索引优化、表结构设计（适当冗余）、SQL 优化（减少 JOIN） |
| 2 | **缓存** | 本地缓存 + 分布式缓存，减少数据库压力 |
| 3 | **分区** | 数据分区减小单表物理存储，提升查询性能 |
| 4 | **数据归档** | 定期清理冷数据到历史表/离线数仓 |
| 5 | **分布式数据库** | TiDB 等，成本高但收益大 |
| 6 | **分库分表** | 最后手段 |

## MySQL 热点数据更新

### 什么是热点数据更新

大促期间，热门商品（如小米旗舰店）被大量用户同时抢购，导致同一行记录被频繁 UPDATE（如库存扣减）。

### 带来的问题

| 问题 | 说明 |
|------|------|
| **锁竞争** | UPDATE 需加排他锁，大量请求阻塞，降低吞吐量 |
| **占用数据库连接** | 阻塞的请求不释放连接，连接池耗尽 |
| **耗尽数据库 CPU** | 大量锁等待 → 自旋 → CPU 消耗 + 线程上下文切换 |
| **死锁风险** | 高并发下频繁定位和更新特定行 |
| **索引维护开销** | 频繁更新导致索引频繁维护 |
| **主从不一致** | 主从复制延迟放大数据不一致概率 |

### 解决思路

- 拆分热点数据（如库存分片）
- 缓存前置（Redis 预减库存）
- 异步削峰（MQ 排队）

## 多级缓存架构

以电商秒杀场景为例，从前到后共 5 层缓存：

| 层级 | 位置 | 内容 | 作用 |
|------|------|------|------|
| 1. 客户端缓存 | 用户设备 | 秒杀倒计时 | 避免每秒请求后端 |
| 2. CDN 缓存 | 边缘节点 | 商品图片等静态资源 | 就近访问，减少源站压力 |
| 3. Nginx 缓存 | 接入层 | 静态资源、用户鉴权（黑白名单） | 过滤非法请求 |
| 4. Web App 本地缓存 | 应用服务器 | 活动时间、用户等级等低频变更数据 | 快速读取，减少远程调用 |
| 5. 分布式缓存 | Redis 集群 | 库存、商品信息等一致性要求高的数据 | 抗高并发读写 |

### 缓存层级选择原则

- **变化越不频繁** → 越靠近客户端（本地缓存）
- **一致性要求越高** → 越靠近分布式缓存
- **性能要求越高** → 越靠近近端缓存

## 常见问题

### Q1：100倍 QPS 突增，如何快速判断是正常流量还是攻击？

看流量来源分布：正常流量通常来自多个渠道、用户行为有规律；DDoS 通常来自少数 IP 段、请求模式单一。结合监控系统的流量来源分析、用户行为分析来判断。

### Q2：为什么数据库优化（索引/SQL）要优先于分库分表？

2000万数据量，用好索引、优化 SQL、减少 JOIN 就能大幅提升性能。分库分表引入跨库事务、分页查询、全局 ID 等复杂问题，改造成本远高于优化成本。
