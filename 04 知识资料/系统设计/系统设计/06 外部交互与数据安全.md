# 外部交互与数据安全

## 一句话结论

与外部机构交互的核心是"防拖垮 + 保安全 + 可对账"：通过异步、超时、限流熔断保护自身，通过加密加签保障数据安全，通过对账机制发现不一致。

## 防止被外部服务拖垮

### 五大策略

| 策略 | 原理 | 适用场景 | 示例 |
|------|------|----------|------|
| **1. 异步交互** | 定时任务/延迟消息异步处理，削峰填谷 | 对实时性要求不高 | 运费险投保（小时级延迟无损） |
| **2. 文件交互** | 批量异步，文件上传FTP对方自行下载 | 大批量、非实时 | 快递单号批量获取 |
| **3. 超时机制** | 设置请求超时，超时即中断 | 必须同步的场景 | 聚合打车叫车（2秒超时） |
| **4. 限流降级熔断** | 限制并发、降级返回兜底、故障时熔断 | 外部接口不稳定 | 用户中心慢→返回"陌生人" |
| **5. 自营替代** | 业务上减少对外部依赖 | 外部SLA不可靠 | 余额宝（减少银行卡支付依赖） |

### 策略选择决策树

```
是否需要实时？
├── 是 → 超时机制 + 限流降级熔断
└── 否 → 异步交互
    └── 量特别大？→ 文件交互
```

## 数据交互安全

### 六大安全措施

| 措施 | 目的 | 实现方式 |
|------|------|----------|
| **协议加密** | 防传输中窃取/篡改 | HTTPS / SFTP / FTPS |
| **数据加密** | 敏感字段保护 | 手机号、身份证号加密 |
| **加签验签** | 完整性 + 身份认证 | 对关键字段哈希后签名，带盐 |
| **身份验证和授权** | 调用者权限控制 | API Key / OAuth / 白名单 |
| **文件完整性校验** | 文件未被篡改 | MD5 校验：文件名+MD5通过接口发送 |
| **区块链** | 不可篡改的共享账本 | 联盟链（金融场景） |

### 加签验签流程

```
发送方：数据 → 哈希 → 用密钥签名 → 附加签名 → 发送
接收方：接收 → 分离数据与签名 → 用密钥验证签名 → 确认完整性
```

## 传输效率优化

| 手段 | 说明 | 优缺点 |
|------|------|--------|
| **专线** | 专用网络连接 | ✅ 高带宽低延迟更安全  ❌ 成本高 |
| **数据压缩** | 减少传输数据量 | ✅ 提升速度  ❌ CPU开销 |
| **URL 代替文件内容** | 图片等上传 OSS，传 URL | ✅ 大幅减少传输量 |

## 数据对账

### 为什么需要对账

分布式系统中，即使使用了各种分布式事务方案，最终一致性方案下系统间调用仍可能失败。需要一套机制来发现不一致。

### 对账分类

| 类型 | 延迟 | 说明 |
|------|------|------|
| **离线对账（D+1）** | 第二天 | D=自然日，数据发生后第二天核对 |
| **离线对账（T+1）** | 下一工作日 | T=交易日 |
| **准实时对账** | 秒级 | 接近实时，但不能保证无延迟 |

### 实现方案对比

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **写代码核对** | 定时任务扫表/拉数据，代码比对 | 通用，支持DB/文件/接口 | 数据量大时性能差，可能OOM |
| **写 SQL 核对** | JOIN/子查询 + WHERE 比对 | 数据库内完成，效率高 | 跨库JOIN受限 |

### SQL 核对执行位置

| 位置 | 适用场景 | 说明 |
|------|----------|------|
| 离线数仓 | 离线核对 | 每天同步数据到数仓，写SQL核对 |
| 在线数据库（备库） | 在线核对 | 避免影响业务，但跨库JOIN受限 |
| 准实时数据库（ADB） | 准实时核对 | 监听binlog同步到ADB，统一空间写SQL |
| ETL 工具 | 数据加工中核对 | 提取→清洗→转化过程中核对 |
| Flink | 流处理核对 | 实时流处理框架 |

### 最佳实践

核对与告警职责分离：
- 核对系统只负责核对
- 告警系统只负责告警
- 设置延迟核对/二次核对/多次失败再告警
- 有告警人工必须跟进，避免无效告警

## 数据库物理删除 vs 逻辑删除

### 为什么很多公司禁止物理删除

| 原因 | 说明 |
|------|------|
| **数据留痕** | 物理删除后数据消失，无法做数据分析/报表/问题排查 |
| **合规要求** | 金融/医疗/法律行业需保留数据供审计 |
| **数据完整性** | 物理删除可能破坏引用完整性 |
| **性能考虑** | 大量物理删除触发数据移动和索引重构 |
| **内存碎片** | InnoDB中delete只标记不释放空间，产生碎片 |

### 逻辑删除实现

```sql
ALTER TABLE t ADD COLUMN is_deleted TINYINT DEFAULT 0;
-- 0=未删除, 1=已删除
```

### 例外：数据归档

定期将历史数据从当前库腾挪到历史库（冷热分离），此时需要物理删除 + 碎片清理。

## 常见问题

### Q1：异步交互和文件交互有什么区别？什么时候用文件？

异步交互是逐条处理（如每单发一个延迟消息），文件交互是批量处理（如每天一个文件）。当数据量特别大、对实时性要求极低（天级别）时用文件交互，可以减少接口调用次数和系统复杂度。

### Q2：数据对账发现不一致后怎么处理？

1. 告警通知开发人员人工介入
2. 设置延迟核对/二次核对避免误报
3. 多次失败后再告警，提高告警阈值
4. 核对系统和告警系统职责分离
5. 人工确认后执行补偿操作
