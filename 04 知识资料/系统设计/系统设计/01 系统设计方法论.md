# 系统设计方法论

## 一句话结论

系统设计需要先做容量估算（QPS、机器数），再根据业务特点做架构选型（多实例 vs 高配），同时关注系统指标的正常范围，形成从需求到方案的完整推导链。

## QPS 估算方法

### 估算流程

| 步骤 | 内容 | 示例 |
|------|------|------|
| 1. 分析业务模型 | 业务类型（B/C端）、用户基数、用户行为 | 微博月活6.05亿，日活≈30% = 2亿 |
| 2. 参考历史数据 | 已有系统分析历史峰值；新系统参考行业标准 | — |
| 3. 估算日均请求量 | 活跃用户 × 人均请求次数 | 2亿 × 2次 = 4亿次/天 |
| 4. 计算峰值流量 | 峰值 = 日均 × 倍数（通常2-3倍） | 峰值3倍 |
| 5. 估算 QPS | 峰值流量 ÷ 高峰时段秒数 | 约12000 QPS（峰值） |
| 6. 预估增长趋势 | 考虑业务增长预留 buffer | 留20%-50% 冗余 |

### 公式推导示例

假设日常 QPS 为 X，高峰时段 2 小时，峰值是日常的 3 倍：

```
X × (24-2) × 3600 + 2 × 3 × X × 3600 = 日均总请求量
```

解得 X ≈ 4000（日常），峰值 ≈ 12000。

## 机器数量估算

### 接口 QPS 与机器数

**已知条件**：接口 QPS=3000，RT=200ms

| 计算步骤 | 公式 | 结果 |
|----------|------|------|
| 单线程吞吐量 | 1000ms / RT | 1000/200 = 5 请求/秒 |
| 单机吞吐量（Tomcat 200线程） | 5 × 200 | 1000 QPS |
| 理论机器数 | 3000 / 1000 | 3 台 |
| 实际机器数（2-3倍冗余） | 3 × (2~3) | 6-9 台 |

> **关键提醒**：以上计算完全理想化，忽略了 CPU、内存、Load 等硬件因素。实际需要在单机压测后根据结果预估。

## 机器选型：4C8G×16 vs 8C16G×8

| 维度 | 4C8G × 16台 | 8C16G × 8台 |
|------|-------------|-------------|
| 并发处理能力 | ✅ 更强（更多实例） | ❌ 较弱 |
| 容错能力 | ✅ 更好（故障影响小） | ❌ 单机故障影响大 |
| 负载均衡 | ✅ 更均匀 | ❌ 粒度粗 |
| 连接数瓶颈 | ✅ 分散 | ❌ 集中 |
| GC 次数 | ❌ 更多（内存小） | ✅ 更少 |
| 单机瓶颈 | ✅ 不易触及 | ❌ 更易触及 |
| 扩容效率 | ❌ 需扩更多台 | ✅ 扩少量即可 |

### 选型建议

- **微服务拆分细、职责简单** → 选多台低配（4C8G×16），容错和负载均衡更重要
- **服务较重、内存/CPU需求高** → 选高配（8C16G×8），减少GC和单机瓶颈

## 4C8G 机器系统指标参考

| 指标 | 正常范围 | 告警阈值 |
|------|----------|----------|
| CPU 使用率 | < 70% | > 85% |
| 内存使用率 | < 80% | > 90% |
| Load Average | < CPU核数 | > CPU核数×2 |
| 磁盘 IO | < 70% | > 85% |
| 网络带宽 | < 70% | > 85% |
| GC 停顿时间 | < 100ms | > 500ms |

## 常见问题

### Q1：为什么实际机器数要比理论计算多 2-3 倍？

理论计算忽略了 CPU 竞争、内存分配、GC 停顿、锁竞争、网络抖动等实际开销。随着并发升高，硬件资源消耗非线性增长，需要留足冗余。

### Q2：新系统没有历史数据，如何预估 QPS？

参考同类产品的公开数据或行业标准，结合目标用户规模和用户行为模型做合理假设。假设值越少，预估越精确。最后一定要留 buffer。
