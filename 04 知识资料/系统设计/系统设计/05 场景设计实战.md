# 场景设计实战

## 一句话结论

秒杀、抢红包、订单管理是面试最高频的场景设计题，核心思路是"逐层流量过滤 + 库存原子扣减 + 异步削峰 + 防刷隔离"。

## 秒杀系统设计

### 七大核心挑战

| 挑战 | 问题 | 解决方案 |
|------|------|----------|
| **1. 高并发瞬时流量** | 海量请求瞬间涌入 | 逐层过滤：客户端随机丢弃 → CDN → Nginx限流 → 服务端Sentinel → 缓存 |
| **2. 热点数据** | 同一商品高频读写 | 拆分（数据分片）+ 缓存预热（Redis + 本地缓存） |
| **3. 数据量大** | 历史数据影响查询 | 缓存 / ES / 分库分表 / 数据归档 |
| **4. 库存正确扣减** | 超卖 / 少卖 | Redis + Lua 原子扣减 → MQ → DB 持久化 + 对账 |
| **5. 黄牛抢购** | 脚本/刷单 | 风控模型（IP/设备/行为）→ 黑名单 → Nginx/业务层过滤 → 限流 |
| **6. 重复下单** | 用户频繁点击 | Token机制 + 限购 + 分布式锁保证幂等 |
| **7. 对普通交易影响** | 秒杀拖垮正常业务 | 隔离：物理隔离（前后端+存储）或逻辑隔离（打标） |

### 库存扣减：避免超卖和少卖

#### 核心原则

**原子性** + **有序性**：库存查询、判断、扣减作为原子操作，并发请求排队执行。

#### Redis + Lua 方案

```lua
-- 原子扣减库存
local stock = redis.call('GET', KEYS[1])
if tonumber(stock) > 0 then
    redis.call('DECR', KEYS[1])
    return 1  -- 扣减成功
else
    return 0  -- 库存不足
end
```

#### 完整链路

```
用户请求 → Redis Lua扣减 → 发送MQ → 消费者 → DB扣减 + 业务处理
```

#### 少卖问题

| 问题 | 原因 | 方案 |
|------|------|------|
| Redis扣减成功，DB未扣减 | MQ丢失/消费失败 | 对账机制：zset流水 + 定时比对 + 补偿 |

> 成熟电商公司中，无论前面方案多完善，核对系统必不可少。

### 业务手段辅助

- **预约制**：预约通过后才能参与秒杀 → 大幅降低请求量
- **限购**：每人限购N件
- **验证码/问题**：降低瞬时流量 + 防脚本
- **预售**：提前锁定需求

## 抢红包设计

### 基本原则

| 原则 | 说明 |
|------|------|
| 公平性 | 每个人都有机会 |
| 随机性 | 金额不可预测 |
| 非零原则 | 每个红包 ≥ 0.01元 |
| 总额不变 | 所有红包之和 = 总金额 |

### 二倍均值法

```
每次红包金额 ∈ [0.01, 剩余金额/剩余个数 × 2]
```

**示例**：100元/10个红包
- 第1个：0.01 ~ (100/10)×2 = 20元 → 随机得6元
- 剩余：94元/9个 → 0.01 ~ (94/9)×2 ≈ 20.89元
- 依次类推，最后不够时保证每人至少0.01元

### 其他考虑

- **性能**：需要快速响应
- **高并发**：优化DB、缓存、网络
- **安全性**：随机性不能被预测或操纵
- **用户体验**：避免极端金额

## 订单号生成服务

| 方案 | 说明 |
|------|------|
| **Snowflake 算法** | Twitter开源，64位：时间戳+机器ID+序列号 |
| **百度 UidGenerator** | 基于Snowflake的改进 |
| **美团 Leaf** | 号段模式 + Snowflake模式 |
| **组合参数** | 时间戳 + 商户编号 + 订单类型 + 商品类型 |
| **缓存预生成** | Redis预生成一批订单号，减少实时生成压力 |

### 设计要点

- 全局唯一 + 趋势递增
- 可配置生成规则
- 分布式锁防重复
- 高并发支持（多线程/异步）
- 失败重试机制

## 订单到期关闭

### 方案全景对比

| 方案 | 推荐度 | 适用场景 | 核心问题 |
|------|--------|----------|----------|
| 被动关闭 | ❌ | 学习用 | 用户不访问就永远不关 |
| **定时任务** | ⭐⭐⭐ | 精度要求不高、业务量大 | 时间不精准、集中扫表 |
| DelayQueue | ❌ | 单机、数据量小 | 内存限制、不持久化、集群难扩展 |
| Netty时间轮 | ❌ | 单机、数据量小 | 同DelayQueue，但效率更高 |
| Kafka时间轮 | ⭐ | 分布式、已用Kafka | 实现复杂 |
| RocketMQ延迟消息 | ⭐⭐ | 延迟时长匹配 | 仅支持固定延迟时长 |
| RabbitMQ死信队列 | ⭐⭐ | 需灵活延迟 | 队头阻塞风险 |
| RabbitMQ插件 | ⭐⭐ | 需灵活延迟 | 插件依赖 |
| Redis过期监听 | ❌ | — | 消息延迟大、可能丢失 |
| Redis ZSet | ⭐ | 分布式 | 可能重复消费 |
| **Redisson** | ⭐⭐⭐ | 分布式、高并发 | 基于ZSet封装，解决重复问题 |

### 复杂度排序（低→高）

被动关闭 < DelayQueue < 时间轮 < 定时任务 < Redis过期监听 ≈ Kafka时间轮 < Redis ZSet ≈ RocketMQ延迟消息 < RabbitMQ死信队列 < RabbitMQ插件 < Redisson

### 选型建议

- **业务量特别大** → 定时任务（最简单可靠）
- **分布式 + 高并发** → Redisson / RabbitMQ插件 / RocketMQ延迟消息
- **精度要求高** → Redisson

## 支付并发问题

### 多渠道同时支付成功

- 支付单冗余 `支付渠道 + 渠道支付单号`
- 回调时先检查状态：
  - 已支付 + 渠道一致 → 幂等成功
  - 已支付 + 渠道不一致 → 重复支付 → 退款

### 订单超时关闭 vs 支付成功并发

**核心原则**：支付成功和已取消都是终态，终态不可再变化。

| 场景 | 处理 |
|------|------|
| 支付成功先到，超时后到 | 超时请求直接拒绝 ✅ |
| 超时先到，支付成功后到 | 触发原路退款 + 资金恒等式校验 |

**资金恒等式**：
- 支付成功：支付金额 > 0，冲退金额 = 0
- 已取消：支付金额 - 冲退金额 = 0

**优化**：进入业务逻辑前先抢分布式锁，降低并发冲突。

## 常见问题

### Q1：秒杀系统为什么要在客户端做随机丢弃？

在离用户最近的地方过滤流量，成本最低。客户端随机丢弃一部分请求直接返回"系统繁忙"，可以减少向后端发送的请求量，保护整个链路。秒杀业务本身就支持失败，用户体验可接受。

### Q2：定时任务扫表关单有什么问题？为什么还推荐？

问题：时间不精准、集中扫表压力大、分库分表后全表扫描不可行。推荐原因：订单到期关闭对精度要求不高（晚几分钟关单影响不大），实现最简单，不需要引入额外中间件。大业务量时可分批扫 + 索引优化。
