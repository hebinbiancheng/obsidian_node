# 面试场景题精选

## 一句话结论

微信红包、扫码登录、单点登录、本地缓存设计是面试中除秒杀外最高频的场景题，考察候选人对真实业务系统的理解深度和技术选型能力。

## 微信红包系统

### 核心设计决策

| 决策点 | 方案 | 原因 |
|--------|------|------|
| 金额计算时机 | **实时计算**（非预分配） | 纯内存计算效率高，预分配需占存储 |
| 金额算法 | **二倍均值法** | 简单高效，拍脑袋算法 |
| 存储 | Redis 集群 | 一条记录存红包信息 + 计数器 |
| 并发控制 | CAS 更新剩余金额和个数 | 原子操作保证一致性 |
| 入账 | 异步操作 | 先抢到再异步入账 |

### 关键问答

**Q：为什么明明抢到红包，点开后发现没有？**
- 2014年：抢和拆是一体的，点开就知道金额
- 2015年后：抢和拆分离，进入第一个页面只表示当时红包还有，不代表抢到

**Q：红包如何判断被抢完？**
- Cache 记录红包个数，原子递减，到0表示抢光
- Cache 过滤无效请求，实际进入后台的量不大

**Q：入账出错怎么办（红包个数没了但余额还有）？**
- 最后有 take all 操作兜底
- 另有对账机制保障

### 微信红包架构简图

```
用户请求 → Cache(Redis)过滤无效请求 → CAS扣减 → 财付通一致性调用
                                              ↓
                              交易记录(双方保存) → 第三方审计
                                              ↓
                              异步入账 → 对账保障
```

## 扫码登录

### 三阶段流程

| 阶段 | 交互方 | 关键动作 |
|------|--------|----------|
| **待扫描** | PC ↔ 服务端 | PC携带设备信息请求生成二维码ID，服务端关联二维码ID与PC设备，PC轮询二维码状态 |
| **已扫码待确认** | 手机 ↔ 服务端 | 手机扫码获取二维码ID，将token+二维码ID发给服务端，服务端关联并生成一次性token |
| **已确认** | 手机 → 服务端 → PC | 手机携带一次性token确认登录，服务端生成正式token给PC，PC完成登录 |

### 关键设计

- 二维码ID = UUID，与PC设备信息关联
- 手机端token + 设备信息绑定（基于token的认证机制）
- 一次性token用于确认凭证（防重放）
- PC端定时器轮询二维码状态变化

## 单点登录（SSO）

### 核心概念

| 概念 | 说明 |
|------|------|
| **认证中心（SSO Server）** | 唯一接受用户名密码，签发授权令牌 |
| **全局会话** | 用户与认证中心之间的会话 |
| **局部会话** | 用户与各子系统之间的会话 |
| **授权令牌** | 认证中心签发，子系统凭此创建局部会话 |

### 会话关系

- 局部会话存在 → 全局会话一定存在
- 全局会话存在 → 局部会话不一定存在
- 全局会话销毁 → 局部会话必须销毁

### 登录流程

```
1. 用户访问系统1 → 未登录 → 跳转SSO认证中心
2. SSO发现未登录 → 引导登录页
3. 用户输入账号密码
4. SSO验证 → 创建全局会话 + 授权令牌
5. SSO携带令牌跳回系统1
6. 系统1拿令牌去SSO校验
7. SSO返回有效 → 系统1创建局部会话
8. 用户访问系统2 → 未登录 → 跳转SSO
9. SSO发现已登录 → 直接携带令牌跳回系统2
10. 系统2校验令牌 → 创建局部会话
```

### 注销流程

```
用户注销系统1 → 系统1通知SSO → SSO销毁全局会话
→ SSO通知所有注册系统 → 各系统销毁局部会话
```

### 部署要点

- 子系统需集成 SSO 客户端
- SSO 认证中心是服务端
- 通信方式：HTTPClient / RPC / RESTful API

## 本地缓存设计

### 设计维度

| 维度 | 考虑点 | 方案 |
|------|--------|------|
| **数据结构** | 用什么存储 | 简单：Map；复杂：多类型（Hash/List/Set等） |
| **对象上限** | 内存有限，存多少 | 设定最大容量（如1024） |
| **清除策略** | 满了怎么办 | LRU / FIFO / LFU / SOFT / WEAK |
| **过期时间** | 数据时效性 | 每个key设置TTL，双重保障 |
| **线程安全** | 并发访问 | ConcurrentHashMap / 读写锁 |
| **接口设计** | 易用性 | get / put / remove / clear / getSize |
| **持久化** | 是否需要 | 可选（如ehcache支持，guava不支持） |

### 清除策略对比

| 策略 | 全称 | 规则 |
|------|------|------|
| LRU | Least Recently Used | 移除最近最少使用的 |
| LFU | Least Frequently Used | 移除使用频率最低的 |
| FIFO | First In First Out | 移除最早进入的 |
| SOFT | Soft Reference | 内存不足时GC回收 |
| WEAK | Weak Reference | GC时立即回收 |

## 接口响应慢排查

### Arthas 排查步骤

| 步骤 | 操作 |
|------|------|
| 1. 发现慢接口 | 通过监控系统确认具体哪个接口慢 |
| 2. 安装 Arthas | `curl -L http://start.alibaba-inc.com/install.sh \| sh` |
| 3. 启动 | 执行启动命令 |
| 4. trace 追踪 | `trace <类名> <方法名>` 查看调用链路耗时 |
| 5. 过滤慢请求 | `trace ... '#cost > 1000'` 只观测超过1000ms的 |
| 6. 分析瓶颈 | 定位耗时最长的方法/代码 |
| 7. 优化 | SQL优化 / 加缓存 / 异步化 / 代码优化 |

### RPC 客户端超时但服务端不超时

| 原因 | 说明 | 排查方法 |
|------|------|----------|
| **网络延迟/丢包**（主要） | 网络传输耗时导致客户端超时 | 查看 TCP 重传率 |
| **超时配置不一致**（主要） | 客户端超时 < 服务端超时 | 检查双方配置 |
| **客户端处理能力**（次要） | 客户端资源瓶颈 | 检查客户端 CPU/内存 |

## 常见问题

### Q1：微信红包为什么用实时计算而不是预分配金额？

实时计算在内存中完成，效率极高。预分配需要额外存储空间存储每个红包的金额，对于海量红包场景存储开销大。且红包有效期只有几天，实时计算完全可行。

### Q2：SSO 的全局会话和局部会话为什么要区分？

职责分离：全局会话由认证中心管理，代表用户的登录状态；局部会话由各子系统独立管理，代表用户在该系统的授权状态。这样认证中心不需要知道每个子系统的业务细节，子系统也不需要重复实现认证逻辑。
