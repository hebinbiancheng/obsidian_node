# 分布式系统核心

## 一句话结论

分布式系统的核心挑战在于协调多个独立节点的一致性、可用性和性能，RPC 框架是通信基础，分布式锁和事务是数据一致性的保障，锁与事务的粒度关系是关键设计决策。

## RPC 框架设计

### 核心模块

| 模块 | 解决的问题 | 技术选型 |
|------|-----------|----------|
| **通信协议** | 客户端与服务端如何通信 | Netty / HTTP2 / 自定义协议 |
| **序列化协议** | 参数如何编码传输 | Protobuf / Hessian / JSON |
| **服务注册与发现** | 服务提供者如何被找到 | ZooKeeper / Nacos / Consul |
| **负载均衡** | 多实例如何选择 | 随机 / 轮询 / 加权 / 一致性哈希 |
| **动态代理** | 像调用本地方法一样调用远程 | JDK Proxy / CGLIB / Javassist |

### 扩展能力

缓存、服务降级、泛化调用、优雅上下线、高可用、异步回调、服务治理。

## 分布式锁 vs 事务粒度

### 核心问题

> 锁的粒度和事务的粒度，谁应该更大？

### 方案对比

| 维度 | 锁粒度 > 事务粒度（先加锁再开事务） | 事务粒度 > 锁粒度（先开事务再加锁） |
|------|--------------------------------------|--------------------------------------|
| **做法** | 加锁 → 开启事务 → 业务 → 提交事务 → 释放锁 | 开启事务 → 加锁 → 业务 → 释放锁 → 提交事务 |
| **事务时长** | ✅ 不受锁影响 | ❌ 事务中包含外部调用（Redis），拖长事务 |
| **锁的时长** | ❌ 跨越整个事务，更长 | ✅ 锁的时间短，并发度高 |
| **数据一致性** | ✅ 安全 | ❌ 可能导致数据不一致 |
| **数据库连接** | ✅ 不额外占用 | ❌ 事务中外部调用占用连接 |

### 建议

**先加锁，再开事务**。虽然锁粒度变大，但：
- 有效避免数据库连接被占用
- 避免数据不一致
- Redis 资源成本低于数据库资源
- 锁通常有续期机制，时长增加可忽略

## 一锁二查三更新为什么还有重复数据

### 问题场景

```text
1. 加锁
2. 查询（是否存在）
3. 更新（不存在则插入）
```

### 根因

**事务粒度 > 锁粒度**：锁在事务提交前就释放了，两个并发事务都查不到数据，都执行了插入。

```
线程A: 加锁 → 查询(无) → 释放锁 → 提交事务(插入)
线程B:          加锁 → 查询(无) → 释放锁 → 提交事务(插入)
                        ↑ A还没提交，B查不到
```

### 解决方案

- **减小事务粒度**：编程式事务，锁内完成查询+更新+提交
- **增大锁粒度**：事务提交后再释放锁
- **使用唯一索引**：数据库层面兜底

## 分布式事务方案选型

### 电商下单场景

5个系统：订单、库存、积分、邮件通知、外部机构（运费险）

| 系统对 | 一致性要求 | 推荐方案 |
|--------|-----------|----------|
| 订单 ↔ 库存 | **强一致性** | TCC / Seata AT |
| 订单 ↔ 积分 | 最终一致性（秒级） | 可靠消息最终一致性 |
| 订单 ↔ 邮件通知 | 弱一致性 | 最大努力通知 |
| 订单 ↔ 外部机构 | 最终一致性（小时级） | 异步文件交互 + 协议约定 |

### 协议约定技巧（外部机构）

1. 只要风控通过，投保一定成功
2. 可以先资金流理赔，再补齐信息流

有了以上约定，就可以把外部交互完全做成异步（离线文件/每日同步）。

## 常见问题

### Q1：为什么事务中不建议做外部调用？

1. **增加事务持续时间**：外部网络调用显著延长事务，占用数据库资源
2. **死锁和资源竞争**：长时间持锁增加死锁概率
3. **提高事务失败风险**：外部调用失败可能导致不必要的回滚（如发MQ失败回滚订单不合理）
4. **难以保证原子性**：外部系统操作不易回滚，超时异常时远程可能已执行成功

### Q2：分布式锁用 Redis 实现有什么注意事项？

- 设置过期时间防止死锁
- 锁的 value 使用唯一标识（UUID），释放时校验身份
- 使用 Lua 脚本保证加锁和释放的原子性
- 考虑 Redisson 的看门狗机制自动续期
- 主从架构下考虑 Redlock 算法
