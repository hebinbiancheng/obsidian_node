# Claude Code 多 Agent 实现机制

> 原文： [面试官皱眉：“你知道 Claude Code 多Agent实现机制吗？” 我：“何止知道？我还看过源码”，他愣了…](https://mp.weixin.qq.com/s/SJ_d8UOR-i3xcXDNozFx6g)  
> 作者：小林coding  
> 整理时间：2026-06-11

## 一句话总结

这篇文章从 Claude Code 的源码视角，拆解了它的多 Agent 体系：**常规 Subagent 负责父子型任务拆分，Fork Subagent 负责缓存友好的“分身”执行，Coordinator 模式负责真正的并行协作**。其核心设计不是简单“多开几个 Agent”，而是围绕隔离、通信、并发、权限和成本做工程化约束。

## 核心结论

- Multi-Agent 的价值在于把复杂任务拆给多个职责清晰的 Agent，降低上下文压力、减少角色混乱，并支持并发。
- Claude Code 里与 Multi-Agent 相关的机制主要有三类：
  - **常规 Subagent**：父 Agent 派子 Agent 处理局部任务。
  - **Fork Subagent**：继承父 Agent 的上下文前缀，利用缓存降低成本。
  - **Coordinator 模式**：主 Agent 退化为协调者，调度多个 Worker 并行工作。
- 工业级 Multi-Agent 系统的关键不是“会派任务”，而是：
  - 子 Agent 与父 Agent 如何隔离；
  - 多 Agent 如何通信；
  - 工具权限如何收敛；
  - 并发任务如何调度；
  - Token 成本如何控制。

## 1. 为什么需要 Multi-Agent

单 Agent 通常是一个“LLM + 工具 + 循环”的结构。它可以自己决定调用什么工具、调用几次，直到完成任务。但在真实项目里，单 Agent 容易遇到三个问题：

1. **上下文膨胀**  
   调研、实现、评审等阶段会消耗大量上下文，容易导致 token 超限或注意力分散。

2. **职责混乱**  
   一个 Agent 同时承担研究员、工程师、评审员等角色，容易在任务过程中跑偏。

3. **无法天然并发**  
   单 Agent 同一时间只能处理一个任务，不能同时调研多个模块或并行验证多个方向。

Multi-Agent 的思路类似“老板带团队”：主 Agent 负责拆任务、看方向、收结果；不同子 Agent 负责调研、实现、评审等具体工作。

## 2. Multi-Agent 的三种常见形态

| 形态 | 说明 | Claude Code 对应机制 |
|---|---|---|
| 父子型 | 主 Agent 遇到子问题时派 Subagent 去处理，完成后返回结果 | 常规 Subagent / Task 工具 |
| 平级协作型 | 多个 Agent 通过共享状态或消息进行协作 | 工程落地较难，状态同步复杂 |
| 主从型 / Coordinator-Worker | 协调者只负责任务分配、结果收集和最终合成，Worker 负责执行 | Coordinator 模式 |

## 3. Subagent 的隔离机制

Claude Code 的 Subagent 隔离主要体现在两个维度：**工具隔离** 和 **上下文隔离**。

### 3.1 工具隔离：给子 Agent 定制工具箱

主 Agent 通常拥有较大的工具集合，但不能原封不动交给子 Agent，否则会带来递归派发、用户交互权混乱、任务状态污染等问题。

Claude Code 通过多层过滤控制子 Agent 的工具权限：

1. **通用黑名单**  
   禁止所有 Subagent 使用某些危险工具，例如：
   - 再次派发 Subagent 的工具；
   - 主动询问用户的工具；
   - 切换规划模式的工具；
   - 停止其他任务的工具。

2. **自定义 Agent 额外黑名单**  
   用户自定义 Agent 通常比内置 Agent 风险更高，因此权限更保守。

3. **后台异步 Agent 白名单**  
   后台 Agent 默认只允许使用明确列出的工具，避免不可控行为。

> 启示：不要假设所有 Agent 都能使用全部工具。Agent 的工具权限应该按角色、来源和运行方式分级控制。

### 3.2 上下文隔离：按字段粒度处理状态

上下文隔离不是简单地“全共享”或“全新建”。

- **全共享的问题**：子 Agent 读取或修改状态后，可能污染父 Agent 的运行视图。
- **全新建的问题**：子 Agent 会丢失必要的外部信号，例如中止信号、任务注册能力等。

Claude Code 的做法是按字段语义逐项处理：

| 状态类型 | 处理方式 | 原因 |
|---|---|---|
| 文件读取缓存 | 克隆一份 | 避免子 Agent 读取文件后污染父 Agent 的文件视图 |
| 全局 UI 状态写入 | 关闭或空操作 | 防止异步 Agent 与主线程抢写全局状态 |
| 后台任务注册 | 保留通路 | 避免子 Agent 启动的后台任务无法被系统回收 |
| Agent ID / 深度 | 新建并递增 | 便于追踪嵌套层级，防止递归失控 |

> 启示：上下文隔离要按字段粒度设计。每个状态都要单独判断：该复制、该共享、该屏蔽，还是该重新生成。

## 4. 父子 Agent 的通信机制

Claude Code 没有采用简单的同步函数调用方式，而是采用**消息驱动**。

### 4.1 为什么不用同步函数调用

如果父 Agent 直接调用子 Agent 并等待返回，会出现两个问题：

- 长任务会阻塞父 Agent，用户交互体验变差。
- 多个子 Agent 并行时，同步调用会让调度逻辑复杂且不稳定。

### 4.2 消息驱动模型

可以把每个 Subagent 理解成一个有“信箱”的独立执行单元：

- 父 Agent 给子 Agent 发消息：往子 Agent 的消息队列追加指令。
- 子 Agent 在自己的循环边界读取消息，并把它作为新的用户消息继续执行。
- 子 Agent 完成后，通过通知机制把结果交回父 Agent。

这种方式的优势：

- 天然异步；
- 支持并发；
- 兼容 Agentic Loop；
- 消息可以持久化；
- 已完成的 Agent 也可以被重新唤醒继续执行。

## 5. Fork Subagent：缓存友好的分身机制

Fork Subagent 适合“需要继承父 Agent 完整上下文，但不想污染父 Agent 主循环”的场景。

它的核心目标是让子 Agent 的 API 请求前缀与父 Agent 保持字节级一致，从而命中缓存，降低成本。

### 5.1 Fork 需要保持一致的内容

| 内容 | 作用 |
|---|---|
| System Prompt | 决定最核心的请求前缀 |
| User Context | 包含项目上下文等动态信息 |
| System Context | 包含系统环境信息 |
| 工具池顺序与定义 | 工具定义会参与请求序列化 |
| 对话历史前缀 | 决定从哪里开始分叉 |

只要其中任何一项发生字节级差异，缓存就可能失效。

### 5.2 Fork 适用场景

适合：

- 基于父 Agent 当前完整上下文生成 PR 描述；
- 做 post-turn 总结；
- 尝试一条替代推理路径；
- 需要“分身”执行但不想污染主循环的任务。

不适合：

- 明确专业分工的任务，例如专门的代码搜索 Agent、评审 Agent；
- Coordinator 模式下的 Worker 调度；
- 需要定制 System Prompt 的独立专家 Agent。

> 启示：成本优化也是架构能力。生产环境里的 Agent 系统不仅要“能跑”，还要“跑得起”。

## 6. Coordinator 模式：真正的并行协作

Coordinator 模式下，主 Agent 不再是全能执行者，而是变成纯协调者。

它主要做三件事：

1. 派 Worker；
2. 收集 Worker 结果；
3. 理解并合成最终答案。

### 6.1 Coordinator 的工作方式

Coordinator 模式强调：**并行是超能力**。

对于可以独立执行的任务，协调者应尽量一次性派出多个 Worker，而不是串行等待。

例如调研一个复杂项目时：

- Worker A 调研认证模块；
- Worker B 调研会话模块；
- Worker C 调研 Token 模块；
- Coordinator 汇总三个结果，形成实现方案。

### 6.2 典型任务流水线

| 阶段 | 执行者 | 目的 |
|---|---|---|
| 调研 | Workers 并行 | 理解代码库、定位问题、收集事实 |
| 合成 | Coordinator | 理解中间结果，制定具体规格 |
| 实现 | Workers | 按规格完成修改 |
| 验证 | 新 Worker 或独立流程 | 用新鲜视角验证结果 |

关键点：Coordinator 不能只当“传话筒”。它必须理解 Worker 的发现，并把发现转化成明确决策和具体规格。

### 6.3 Continue vs Spawn

协调者要判断：是继续使用已有 Worker，还是创建新 Worker。

| 场景 | 推荐做法 | 原因 |
|---|---|---|
| 新任务与旧 Worker 上下文高度相关 | Continue | 复用上下文，减少沟通成本 |
| 新任务与旧上下文无关 | Spawn | 避免旧上下文干扰 |
| 旧 Worker 已经跑偏 | Spawn | 防止错误方向延续 |
| 验证任务 | Spawn | 保持新鲜视角，避免自己验证自己 |

## 7. 常规 Subagent 与 Coordinator 模式对比

| 维度 | 常规 Subagent | Coordinator 模式 |
|---|---|---|
| 主 Agent 角色 | 全能执行者，必要时派助手 | 纯协调者 |
| 子 Agent 执行方式 | 偏父子结构 | 默认异步 Worker |
| 并发程度 | 偶尔并发 | 最大化并发 |
| 适合场景 | 单个任务中的局部帮手 | 大任务、高并发拆解 |
| 系统形态 | 父子树 | 协调者 + Worker 扁平层 |
| 风险控制 | 子 Agent 工具隔离 | Worker 禁止继续协调别人，防递归扩散 |

## 8. 可迁移的 5 条 Multi-Agent 设计原则

### 原则 1：上下文隔离要按字段粒度做

不要粗暴地全共享或全新建。对每个状态分别判断：是否会污染父 Agent、是否需要被子 Agent 感知、是否需要独立副本。

### 原则 2：通信走消息，不走同步函数调用

消息队列更适合 Agentic Loop，也更容易支持异步、并发、持久化和任务恢复。

### 原则 3：工具权限要分级管控

不同 Agent 应该拥有不同工具集。尤其要防止子 Agent 继续派子 Agent，导致递归失控。

### 原则 4：缓存友好是一种架构能力

在高频、多 Agent 场景下，prompt 前缀复用和缓存命中会显著影响成本与延迟。

### 原则 5：并行优先，协调者负责合成

真正的 Multi-Agent 价值来自并行。但并行结果必须由协调者理解、决策和合成，而不是简单转发。

## 我的理解

这篇文章最值得沉淀的点是：**Multi-Agent 的本质不是多几个 LLM 实例，而是围绕任务拆解、状态隔离、权限收敛、异步通信和结果合成建立一套可控的协作系统。**

如果把这个思路迁移到自己的 Agent 系统，可以优先检查五个问题：

1. 子 Agent 是否会污染父 Agent 状态？
2. 子 Agent 是否拿到了过多工具权限？
3. 父子 Agent 通信是否会阻塞主循环？
4. 是否能并行派发互不依赖的任务？
5. 协调者是否真正理解并合成结果，而不是转发结果？
