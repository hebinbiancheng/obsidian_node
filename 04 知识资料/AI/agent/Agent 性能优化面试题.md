# Agent 性能优化面试题

> 原文：[微信文章](https://mp.weixin.qq.com/s/anLpe2uUqGl_2wfe0cZlTw) · 2026-07-17
> 原始资料：`^[raw/articles/wechat-agent-performance-2026.html]`

---

## 一句话总结

Agent 性能优化 = 降 Token（精简 Prompt、会话摘要、Rerank、小模型分流）+ 提速度（意图识别前置、流式输出、工具并行、缓存分层）+ 可观测（四层监控、链路追踪、故障定位七环节）。

---

## 1. 如何降低 Token 成本

| 方向 | 做法 |
|------|------|
| **精简 System Prompt** | 去掉重复/无意义提示词；规则校验用代码不用 Prompt |
| **减少无关上下文** | 只放与当前问题最相关的部分，不全量塞历史 |
| **会话摘要** | 早期聊天内容摘要压缩后再入 Prompt |
| **RAG Rerank 重排序** | 先多召回，Rerank 后只取最相关的文档片段 |
| **限制输出长度** | 明确要求只输出答案，不输出分析过程 |
| **不同模型分流** | 复杂分析→GPT/Claude；分类/意图识别→小参数模型 |

---

## 2. 如何提高 Agent 响应速度

| 方向 | 做法 |
|------|------|
| **意图识别前置** | 简单问题直接 LLM 回答，不跑全流程 |
| **减少 Prompt 长度** | 精简提示词 + 历史 + RAG 文档 |
| **流式输出** | 不仅 LLM 流式返回，记忆召回/工具调用/RAG 也流式展示 |
| **工具并行执行** | 不互依赖的工具并发调用 |
| **热点数据缓存** | Embedding 向量、热点 RAG 结果、长期记忆 |
| **工具超时** | 设置超时，不无限期等待 |

---

## 3. Agent 应用中哪些部分可以用缓存

| 缓存类型 | 内容 | 注意事项 |
|---------|------|---------|
| **回答缓存** | 完全相同提问的最终答案 | 按用户隔离 Key |
| **RAG 检索缓存** | 热点问题的检索结果 | 设置合理过期时间 |
| **Embedding 缓存** | 同一文本的向量 | 按文本 hash 去重 |
| **工具结果缓存** | 天气/汇率/商品等信息查询 | 按业务设置 TTL |
| **用户画像缓存** | 偏好、组织信息等高频读取 | 按 user_id 隔离 |

---

## 4. Agent 四层监控

| 层 | 指标 |
|----|------|
| **接口层** | 请求量、成功率、错误率、平均响应时间 |
| **LLM 层** | 调用次数、首 Token 时间、总响应时间、输入/输出 Token 量 |
| **工具层** | 调用次数、成功率、重试次数、平均耗时、失败原因 |
| **业务层** | 任务完成率、点赞/点踩、转人工率 |

技术栈：Prometheus + Grafana（指标）、ELK（日志）、OpenTelemetry（链路）、LangSmith（LLM/工具节点追踪）。

---

## 5. Token 消耗统计

每次 LLM 调用的响应都返回：输入 Token、输出 Token、总 Token。累加一次任务中所有调用：

```
本次调用费用 = 输入Token × 单价 + 输出Token × 单价
```

**注意**：本地分词器适合**估算**，不适合**账单计算**——不同模型厂商的 Token 计算方法不同。缓存命中时按缓存单价计算。

---

## 6. Tool 调用成功率

区分两层：

| 维度 | 含义 |
|------|------|
| **技术成功** | 工具正常返回结果 |
| **业务成功** | 结果表示业务也执行成功（查到订单/创建商品） |

```json
{"trace_id": "trace_001", "tool_name": "search_docs",
 "success": true, "duration_ms": 320, "error_type": null, "retry_count": 0}
```

用 Prometheus + Grafana 做指标监控。

---

## 7. 如何定位 Agent 故障（七环节排查）

```
1. 用户输入 → 是否完整、有歧义
2. Prompt → 是否正确，记忆/RAG 是否相关
3. LLM 推理 → 选对工具了吗，参数对吗
4. RAG 检索 → 文档正常召回？语义完整？重排正确？
5. 工具执行 → 超时/限流/权限/参数校验/接口异常
6. 状态流转 → 是否进错分支、重复执行、超最大步骤
7. 最终回答 → 是否正确使用工具结果，是否有幻觉
```

定位后复现：Agent 输出不确定性高，需多次执行判断是必现还是偶发。

---

## 8. Agent 链路追踪

一次用户请求 → 唯一 Trace ID → 内部各步骤（LLM/RAG/工具/记忆召回）生成 Span：

```json
{"trace_id": "trace_001", "span_id": "span_002",
 "name": "search_docs", "type": "tool",
 "start_time": "...", "end_time": "...", "duration_ms": 320,
 "status": "success", "input": {}, "output": {}, "error": null}
```

LLM span 额外记录：模型名称、Token 用量。RAG span 额外记录：召回片段 ID、得分。

实现：LangSmith 或 OpenTelemetry + Prometheus + Grafana。

---

## 9. 模型不可用处理

| 层级 | 做法 |
|------|------|
| **超时 + 重试** | 限制重试次数，指数退避 + 随机抖动 |
| **Fallback** | 主模型不可用 → 自动切备用模型 |
| **熔断** | 同一模型连续失败 → 停止调用，避免故障扩大 |
| **降级** | 返回「当前不可用，请稍后重试」 |
| **可观测** | Trace 记录错误类型/失败模型，Metrics 监控成功率/延迟 |
