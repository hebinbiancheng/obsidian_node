# Agent 主循环终止条件：凭什么知道该停

> 原文：[微信文章](https://mp.weixin.qq.com/s/PWgYUtWB07YKyMFGxXSjMA) · 2026-06-28
> 原始资料：`^[raw/articles/wechat-article-*.html]`

---

## 一句话总结

Agent 主循环的本质是「只要模型还在调工具就继续」——模型没有完成开关，停止逻辑必须由工程侧的三层机制（指纹去重 + 结构化终止 + 硬上限兜底）来保证。

---

## Round 1：为什么 Agent 会卡在一步上反复试？

> 同一个工具、同样的参数调了十几次——不是 bug，是默认行为。

**根因**：Agent 主循环的默认逻辑就是「模型在调工具 → 继续循环」。模型每一轮只是概率生成下一个动作，它没有「我卡住了」的内置开关。

**学术定义**（arxiv 2601.15679）：同一工具 + 完全相同参数，连续 **3 轮以上** → 判定为逻辑死循环。

**常见触发点**：工具返回 `None`、空字符串或生硬报错。模型当成没拿到结果，原样重试。

**自检方法**：
1. 打开 Agent trajectory log，数连续相同的工具调用
2. 查框架默认循环上限——Vercel AI SDK 默认 `stepCountIs(1)`（压根不循环！）

### 关键速记

- 模型没有「完成与否」的内置开关
- 死循环 = 同一工具 + 同样参数连续 3 轮以上
- 触发源：工具返回 None/空/报错
- Vercel AI SDK 默认不循环，LangGraph 靠 `recursion_limit` 硬截断

## Round 2：Agent 凭什么判断任务做完了？

> 候选人说「模型自己会判断」——只说对了三分之一，而且是其中最不可靠的那种。

**Agent 判停只有三条路，没有一条靠模型真的知道：**

### 1. 模型不再调工具 → 自然结束（最常用、最不可靠）

OpenAI Agents SDK 做法：runner 循环，有工具调用就执行，直到某轮模型只输出答案不再调工具。

问题：「要不要再调工具」同样是概率生成，模型对「目标达成」没有确定性感知——它只是觉得这里像是该收尾了。

### 2. 显式 `done` 工具

定义一个只用来宣布完成、本身不执行的 done 工具。模型调用它，循环停。

本质：把「停」也包装成一个动作，塞进选项里让它选。

### 3. 完成度评估器

`evaluate_completion(结果, 目标)`，绑定可观测状态变化：工单关了没？文件写了没？消息发出去没有？

关键：把「做完了」定义为真实世界里能查到的**状态跳变**，而不是模型嘴上说的。

> ⚠️ 让模型自己决定何时停，根本不是一个策略。基础推理循环本身不保证终止。

**三个放大器**（凑齐就鬼打墙）：目标模糊 + 工具反馈含糊 + 自我条件化（上一步错误在上下文中原封不动）

## Round 3：生产环境怎么保证一定停？

> 候选人说「加个 max_iterations」——硬上限是安全网，不是解法。

生产用 **三层叠加**：

### 第一层：动作指纹去重（最简单有效）

```python
from hashlib import md5
from collections import deque

recent = deque(maxlen=20)           # 滑动窗口：最近 20 次调用

def guard(tool, args):
    fp = md5(f"{tool}:{args}".encode()).hexdigest()
    recent.append(fp)
    if recent.count(fp) >= 3:       # 同一指纹 3 次 = 死循环
        return "BLOCKED: 换个思路或宣布完成"
    return None                     # 放行
```

⚠️ 精确哈希只抓一模一样的重复。模型会搞变体循环（`search(认证错误)` → `search(auth报错)` → `search(登录失败)`），需要语义相似度判断。

### 第二层：明确的终止条件

工具返回结构化状态（`ok` / `no_results` / `error`），不要裸字符串。`SUCCESS：订单 HT79265 已确认` 比空 list 好百倍。

### 第三层：硬上限兜底

| 上限类型 | 建议值 | 原因 |
|----------|--------|------|
| max turns | 25 | Agent 烧 token 是普通对话 4 倍，多 Agent 15 倍 |
| max time | 5 分钟 | 超时优雅退出 |
| token 预算 | 按任务设 | 防止账单爆炸 |

**检测到死循环后的分级降级**（来源：OpenClaw 分级策略）：

1. 注入 reflection 提示逼它重新评估
2. 提示换工具/换思路
3. 压缩上下文重启推理
4. 带部分结果优雅退出

> 🔑 关键经验：光注文字警告没用（LLM 会无视），真正拦得住的是代码层硬 halt。

## Round 4：模型更强就能自己学会停吗？

> **不能。这不是模型强度问题，是架构边界。**

判断任务真做完了需要**对世界状态的确定性核验**：工单关了？文件写了？钱打了？——这是查得到、不容狡辩的事实。

模型能做的永远只是「觉得这里像是完成了」，它读的是上下文，不是真实世界状态。

**线上常踩的坑**：下单 Agent 调支付接口，接口超时悄悄失败，返回文本里没有明确失败信号。模型扫一眼上下文，写出「已为您完成下单」。任务在它眼里成了，在真实世界里没成。

> 文本上像完成 ≠ 世界里真完成。再强几代的模型也跨不过这道沟。

**停止权必须攥在代码里**：done 工具、状态机、完成度评估——本质都是把判停权从模型手里拿回工程手里。学术界也在走这个方向，Focused ReAct（arxiv 2410.10779）加了早停，一旦检测到重复动作就强制模型基于现有信息收尾。

## 🎯 面试官点评

能把 Agent 主循环终止条件想清楚的人，做出来的 Agent 不一样：

- 循环里默认挂指纹去重和硬上限
- 工具返回带明确状态（ok/no_results/error）
- 完成绑定到可查的状态跳变，不赌模型突然懂事

**三个立即行动**：
1. 打开 Agent 日志数重复调用，确认框架默认上限
2. 给每个工具返回值加结构化状态
3. 把退出条件、指纹去重、硬上限当成主循环标配，第一版就写进去

> Agent 让它动起来很容易——一个 while 循环就够。难的是让它在该停的时候停下来。这部分不归模型管，归你。
