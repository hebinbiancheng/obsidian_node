三、逐题答案
第 01 题
什么是 ReAct？它的思考-行动-观察循环和工具调用流程是怎样的？
分类：Agent / LLM ｜ 难度：进阶 ｜ 高频公司：淘天、腾讯、字节、百度
一句话：ReAct = 大模型边想边做，工具返回结果再反过来修正下一步。
ReAct 是 Reasoning + Acting 的缩写，本质是让大模型不是一次性直接给答案，而是在任务执行过程中交替进行“思考、行动、观察”。思考用于分析当前目标和下一步，行动用于调用工具或执行外部操作，观察用于接收工具返回结果，然后模型基于观察继续更新判断。
典型循环是：用户问题 → LLM 生成 Thought → 选择 Action 和参数 → 调用工具 → 得到 Observation → LLM 根据 Observation 决定继续调用工具还是输出 Final Answer。
工程实现上，ReAct 不应该完全依赖模型自由发挥。生产系统通常会把工具列表、工具 schema、权限、超时、重试、最大轮数、终止条件都放到 Runtime 里控制。模型只负责选择工具和生成参数，真正执行由调度器完成。
面试回答可以强调：ReAct 适合信息不完整、需要边查边推理的场景，比如 RAG 检索、故障排查、代码分析、数据库问数。它的风险是循环过长、工具幻觉、错误观察污染后续推理，所以需要工具白名单、参数校验、最大步数、观测日志和失败兜底。

回答时先讲定义，再讲流程或架构，最后补一句工程风险与治理手段，会比只背概念更像做过项目。
第 02 题
RAG 中长文档切片的粒度和分块策略如何设计与优化？
分类：RAG ｜ 难度：进阶 ｜ 高频公司：快手、字节、淘天
一句话：先按语义结构切，再按 token 兜底，用评测指标反推最佳粒度。
长文档切片的核心目标不是把文本切得越细越好，而是在“召回准确性、上下文完整性、生成成本、检索速度”之间做平衡。chunk 太小容易丢失上下文，chunk 太大又会降低向量表达精度，并挤占模型上下文。
常见策略是按文档结构优先切分：标题、章节、段落、列表、表格、代码块优先保持完整；再用 token 长度兜底。比如普通知识文档可以 300～800 tokens 一个 chunk，技术手册或法律条款可以偏大，FAQ 可以偏小。
优化上要做 overlap，一般 10%～20%，解决跨段语义断裂；同时保留 metadata，例如文档 ID、标题层级、页码、时间、租户、权限标签。检索结果不只是 chunk 正文，还要带来源，方便生成答案时引用和溯源。
工程上建议做离线评测：固定问题集，比较不同 chunk size、overlap、父子分块、rerank 后的 Recall@K、MRR、答案准确率和成本。不要凭感觉调参数。

第 03 题
RAG 系统评测的维度和常用指标如何设计？
分类：RAG / Evaluation ｜ 难度：进阶 ｜ 高频公司：快手、米可
一句话：RAG 评测 = 检索有没有找对 + 生成有没有答对 + 系统能不能稳定低成本跑。
RAG 评测不能只看最终回答，要拆成三个层面：检索质量、生成质量、系统工程质量。因为答案错可能是没召回，也可能是召回了但模型没用，也可能是模型编了。
检索侧常用指标包括 Recall@K、Precision@K、MRR、nDCG、Hit Rate。关注的是：正确文档是否进入候选集、排序是否靠前、TopK 是否够用。
生成侧常用指标包括答案正确性、忠实性 Faithfulness、上下文相关性 Context Relevance、答案相关性 Answer Relevance、引用准确率、幻觉率。尤其要看答案是否被检索内容支撑。
工程侧要评估延迟、吞吐、成本、索引更新时效、权限过滤正确率、失败率、用户满意度。生产级 RAG 一定要同时做离线评测和线上反馈闭环。

第 04 题
RAG 系统中的文档分块策略如何设计？chunk 大小、父子分块和不同文档格式要怎么取舍？
分类：RAG ｜ 难度：进阶 ｜ 高频公司：Moka、阿里、百度
一句话：子块负责“找得准”，父块负责“答得全”。
chunk 大小取决于文档类型和问题类型。FAQ、短知识点适合小 chunk；制度文档、合同、技术方案适合中等 chunk；表格和代码要尽量保持完整单元，不能粗暴按长度截断。
父子分块适合长文档。子 chunk 用于向量检索，粒度较小，保证召回精准；父 chunk 用于答案生成，包含更多上下文，避免模型只看到片段。比如子块 300 tokens，父块 1000～1500 tokens。
不同格式要采用不同解析策略。PDF 要处理页眉页脚、分页断句、表格丢失；Word 要保留标题层级；Markdown 可以按标题切；Excel 要按 sheet、表头、行组组织；代码文档要按类、函数、注释切。
面试中可以说：我不会只配置一个全局 chunk size，而是按文档类型做 splitter 策略，并通过召回率、答案准确率和成本做 A/B 测试。

第 05 题
多 Agent 协作有哪些常见模式、适用场景和主要挑战？
分类：Agent ｜ 难度：进阶 ｜ 高频公司：字节、阿里云、阿里、蚂蚁、小红书、淘天
一句话：多 Agent 不是多人聊天，而是有角色、有状态、有裁决的任务协作系统。
常见模式有三类：流水线模式、专家协作模式、辩论/评审模式。流水线适合固定流程，比如需求分析 → 编码 → 测试；专家协作适合不同能力拆分，比如检索 Agent、SQL Agent、总结 Agent；评审模式适合高风险任务，比如代码生成后由 Reviewer Agent 检查。
多 Agent 的价值是把复杂任务拆成多个角色，降低单个 Agent 的上下文压力和职责混乱。但不是 Agent 越多越好，多 Agent 会带来调度复杂、成本上升、延迟增加、冲突难收敛的问题。
工程设计上要有统一 Runtime：任务状态机、消息协议、上下文隔离、共享 Memory、工具权限、失败重试、最大轮数、日志追踪。不能让多个 Agent 自由聊天，否则很容易失控。
面试中可以结合项目回答：比如编码 Agent 负责修改文件，计划 Agent 负责拆解任务，批判 Agent 负责审核风险，最终由 Orchestrator 统一裁决。

第 06 题
Agent 的上下文管理和记忆机制通常如何设计，如何避免上下文过长或信息污染？
分类：Agent ｜ 难度：进阶 ｜ 高频公司：快手、百度、字节、阿里云、蚂蚁
一句话：上下文解决“当前任务怎么做”，记忆解决“以后还要不要记住”。
Agent 的上下文一般分为短期上下文、任务状态、长期记忆、工具观察结果。短期上下文保存当前会话，任务状态保存执行进度，长期记忆保存稳定事实或经验，工具观察结果保存可追溯证据。
避免上下文过长的方法包括：滑动窗口、摘要压缩、按需检索、重要信息 pin 住、旧消息归档、结构化状态替代原始对话。不要把所有历史消息都塞进 prompt。
避免信息污染的关键是记忆写入要有门槛。不能模型说一句就写长期记忆，要判断是否长期有效、是否和任务相关、是否可信、是否敏感。记忆读取也要带时间、来源、置信度和适用范围。
工程上可以设计 Memory Pipeline：捕获候选记忆 → 规则过滤 → LLM 总结 → 去重合并 → 打标签 → 写入向量库/数据库 → 使用时按任务召回并重新校验。

第 07 题
在开发工作中如何使用 AI 工具辅助编程和提效？
分类：AICoding / OpenEnded ｜ 难度：进阶 ｜ 高频公司：小红书、蚂蚁、影石
一句话：AI 编程最适合做“副驾驶”，不是替代工程判断。
AI 编程提效不是让模型直接替你写完，而是把它嵌入研发链路：需求拆解、代码阅读、方案对比、样板代码生成、单测生成、日志分析、SQL 排查、文档生成和 Review 辅助。
我的使用方式会分层：简单重复代码让 AI 生成；复杂逻辑让 AI 先给方案和边界条件；涉及线上风险的代码必须人工 Review；数据库变更、权限、删除操作必须加确认。
在团队落地时，要沉淀 Prompt 模板和工程规范，比如：给模型项目背景、接口约束、异常处理要求、代码风格、不能改哪些文件。这样效果比随便问一句要稳定得多。
面试可以强调结果：减少查资料时间、提升代码阅读效率、补充单测、快速定位报错，但最终责任仍然在人，不能盲信模型。

第 08 题
大模型长上下文场景下，如何进行上下文压缩与优化？
分类：LLM / RAG / Agent ｜ 难度：进阶 ｜ 高频公司：快手、淘天、拼多多、腾讯
一句话：上下文优化不是压得越短越好，而是让关键证据优先进入窗口。
长上下文优化的目标是把最有用的信息放进模型窗口，而不是盲目扩大上下文。上下文越长，成本、延迟和注意力噪声都会增加。
常见方法包括：摘要压缩、检索式上下文、层级摘要、关键信息抽取、去重、按角色分区、按时间衰减、上下文打分排序。RAG 场景可以先检索再压缩，Agent 场景可以把历史操作变成结构化状态。
压缩时要避免语义扭曲。高风险信息不建议只保留自然语言摘要，最好保留原文引用、来源 ID 和关键字段。比如告警排查中，原始错误日志、时间戳、指标值不能被摘要改写。
工程实现可以做 Context Builder：先收集候选上下文，再按 relevance、recency、authority、token budget 排序，最后拼成 prompt。

第 09 题
在 LLM 或 RAG 应用中，如何减少和规避幻觉问题？
分类：LLM / RAG ｜ 难度：进阶 ｜ 高频公司：阿里云、京东、蚂蚁
一句话：降低幻觉靠证据约束和工程校验，不是靠一句 prompt。
幻觉的本质是模型在证据不足时仍然生成看似合理的内容。RAG 中常见原因包括召回不到、召回错、上下文过长被忽略、prompt 没约束、模型过度补全。
减少幻觉要从链路治理：检索侧提高召回和重排质量；生成侧要求答案必须基于上下文；证据不足时明确回答“不确定”；工具侧做 schema 校验和错误返回；展示侧提供引用来源。
还可以增加 answer verification：让模型检查答案是否被 context 支撑，或者用规则校验数字、时间、实体、权限。对高风险领域可以加人工审核或只输出候选建议。
面试中要避免只说“调 prompt”。更好的回答是：数据、检索、重排、生成、校验、权限、监控一起治理。

向量检索通常是双塔模型，query 和 document 分别编码成向量，再算相似度。它速度快，适合从海量文档里召回候选，但表达的是粗粒度语义相似。
交叉编码器会把 query 和候选文档一起输入模型，让模型直接判断二者相关性。它更慢，但能捕捉更细的词义、否定、条件、实体关系和上下文匹配。
所以典型链路是：向量检索或混合检索先召回 TopN，比如 50～200；Rerank 再精排到 TopK，比如 5～10；最后把最相关内容交给生成模型。

第 11 题
如何评估 Rerank 的有效性，常用哪些指标？
分类：RAG / Evaluation ｜ 难度：进阶 ｜ 高频公司：快手
一句话：Rerank 好不好，看正确证据有没有更靠前，而不是只看分数变没变。
Rerank 的有效性要看它是否把真正相关的 chunk 排到了更靠前的位置，而不是只看最终大模型回答。
常用离线指标包括 MRR、nDCG@K、Recall@K、Precision@K、Hit Rate、Top1 Accuracy。MRR 看第一个正确结果的位置，nDCG 适合多级相关性，Recall@K 看正确材料是否进入最终上下文。
线上指标可以看答案采纳率、用户追问率、无答案率、引用点击率、平均延迟、成本变化。因为 Rerank 会增加耗时，所以要同时看效果收益和延迟成本。
评测方式是准备 query、标准答案、标准相关文档集合，对比“无 rerank”和“有 rerank”的排序变化及最终答案质量。

第 12 题
GRPO 和 PPO 的核心区别是什么？在信用分配、优势估计和方差控制上分别有什么差异？
分类：LLM / Evaluation ｜ 难度：深入 ｜ 高频公司：作业帮、腾讯、三七互娱
一句话：PPO 用价值模型估优势，GRPO 用同题多答案的组内相对奖励估优势。
PPO 是强化学习中常用的策略优化方法，通常需要策略模型、参考模型、奖励模型和价值模型。它用价值函数估计 advantage，并通过 clip objective 限制策略更新幅度，避免训练不稳定。
GRPO 可以理解为 PPO 的一种变体，常见于大模型推理训练场景。它不再依赖单独的 value model，而是对同一个问题采样一组输出，用组内奖励的相对高低来估计 advantage。
信用分配上，PPO 依赖 value model 对 token 或 trajectory 估值；GRPO 用 group 内多个候选答案的 reward 做相对比较，让“比同组平均更好”的输出获得正优势。优势估计上，PPO 是 value baseline，GRPO 是 group relative baseline。
方差控制上，PPO 的 baseline 来自训练出来的价值模型，效果依赖 value 质量；GRPO 通过同题多样本的组内归一化降低方差，同时省掉 value model 的显存和训练复杂度，但需要多采样，推理采样成本更高。

第 14 题
ReAct 与 Plan-and-Execute 有什么区别，分别适用于什么场景，为什么在项目中会选择 ReAct？

第 18 题
Rerank 的 Top-k 数量通常如何确定？
分类：RAG ｜ 难度：进阶 ｜ 高频公司：快手
一句话：TopK 不是拍脑袋，是由召回曲线、准确率、延迟和成本共同决定。
Rerank 的 Top-k 一般分为两个 K：召回给 Rerank 的 TopN，以及 Rerank 后进入生成模型的 TopK。TopN 通常取 50～200，TopK 通常取 3～10，但必须结合业务评测确定。
TopN 太小会导致正确文档还没进入重排阶段；TopN 太大则增加 Rerank 延迟和成本。TopK 太小会漏上下文，TopK 太大又会让生成模型被噪声干扰。
确定方法是用标注集画曲线：Recall@N 随 N 增长到哪里收益变小，生成答案准确率随 K 变化在哪里最好，同时看延迟和 token 成本。
工程上还可以动态 TopK：如果前几条分数明显高，就少给；如果分数接近或问题复杂，就多给。

一句话：传统 RAG 是固定检索链路，Agentic RAG 是会决策、会拆解、会多轮检索的 RAG。
传统 RAG 通常是固定链路：用户问题 → 检索 → 拼上下文 → 生成答案。它适合知识问答，但流程相对静态。
Agentic RAG 引入 Agent 决策能力，让模型可以判断是否检索、检索哪些库、是否改写 query、是否调用工具、是否二次检索、是否验证答案。流程是动态的。
比如用户问一个复杂问题，Agent 可以先拆解子问题，分别检索文档、查询数据库、调用计算工具，然后汇总答案。这比单次检索更适合复杂任务。
风险是成本和不可控性更高，所以要限制最大步骤、工具权限、检索范围和终止条件。

分类：Agent / LLM ｜ 难度：进阶 ｜ 高频公司：淘天

第 37 题
请详细讲解 ReAct 框架的原理。
分类：Agent ｜ 难度：进阶 ｜ 高频公司：未知
一句话：ReAct 用外部观察纠正内部推理，让 Agent 不再只靠模型记忆回答。
ReAct 的核心思想是把推理 Reasoning 和行动 Acting 交织在一起。传统 CoT 更偏内部推理，工具调用更偏外部行动，ReAct 把二者放进一个循环，让模型可以边推理边获取新信息。
原理上，模型每一轮先根据任务和历史观察生成下一步思路，再选择工具或动作，工具返回观察结果后，模型更新状态。如果问题已经解决，就输出最终答案。
它提升效果的原因是：推理可以帮助模型规划工具使用，行动可以让模型获得外部事实，观察可以纠正错误假设。比如不知道某个接口状态，就调用监控工具，而不是猜。
工程实现时通常不暴露完整思考链给用户，而是在内部保存决策摘要、工具调用和观察结果。对外展示可以是“我查询了哪些信息，基于哪些证据得出结论”。

如何让 Agent 具备自我学习和经验沉淀的能力？
分类：Agent ｜ 难度：深入 ｜ 高频公司：未知
一句话：自我学习不是现场改模型，而是把经验转成可检索、可评测、可复用的资产。
Agent 的自我学习在工程上通常不是让模型实时训练自己，而是把执行经验沉淀到 Memory、案例库、规则库、评测集和 Prompt/Skill 中。
一次任务完成后，可以记录目标、步骤、工具调用、失败原因、最终结果和人工反馈。成功案例沉淀为 SOP，失败案例沉淀为反例和测试集。
记忆写入要有审核机制，不能所有内容都写入。可以只沉淀稳定经验、用户确认过的偏好、可复用操作流程和已验证问题解决方案。
长期看，可以用这些轨迹数据做 SFT、偏好优化或工具调用优化，让模型逐渐学会更稳定的决策。

第 42 题
Rerank 之后的 TopK 截断如何实现，截断值怎么确定？
分类：RAG ｜ 难度：进阶 ｜ 高频公司：快手
一句话：TopK 截断本质是证据选择，要在完整性和噪声之间平衡。
Rerank 后会得到候选 chunk 的相关性分数，TopK 截断就是按分数排序后选择前 K 个进入生成上下文。实现上很简单，难点在 K 怎么定。
固定 K 适合稳定场景，比如总是取前 5 个；动态 K 更灵活，可以根据分数阈值、分数差距、token budget 和问题复杂度决定。
比如设置最低相关分，如果低于阈值就不进入上下文；如果第一名和后面差距很大，只取少量；如果多个 chunk 分数接近且来自不同章节，可以多取。
确定截断值要通过评测，看不同 K 下的答案准确率、幻觉率、上下文 token、延迟和成本。K 不是越大越好，过多噪声会干扰生成。

第 43 题
什么是 Memory？短期记忆在对话轮次增加时如何持续处理上下文爆炸和重复压缩？
分类：Agent ｜ 难度：深入 ｜ 高频公司：阿里、Moka
一句话：短期记忆要从聊天记录演进成结构化任务状态，不能无限堆消息。
Memory 是 Agent 对历史信息的管理机制，不等于简单聊天记录。它包括短期记忆、长期记忆、任务状态和经验知识。
短期记忆用于当前会话连续性，但随着轮次增加会产生上下文爆炸。处理方式包括滑动窗口、增量摘要、重要信息 pin、重复内容去重、按任务阶段归档。
重复压缩会带来语义漂移，所以不能每轮都把摘要覆盖掉。更好的做法是保留结构化状态和关键原文引用，对摘要做版本管理，重要事实单独保存。
当对话进入新阶段时，可以把旧阶段压缩成阶段摘要；当用户回到旧问题时，再按需检索原始记录或长期记忆。

第 44 题
Agent 的 Planning 模块有哪些主流实现方式，Step-by-step 和 Plan-and-execute 有什么区别？
分类：Agent ｜ 难度：进阶 ｜ 高频公司：未知
一句话：Step-by-step 重在即时推进，Plan-and-Execute 重在全局规划和可控执行。
Planning 常见方式包括：隐式逐步推理、显式步骤列表、Plan-and-Execute、任务树/DAG、反思式重规划、人类确认计划。
Step-by-step 通常是模型边做边给出下一步，适合短任务和不确定场景；Plan-and-Execute 是先规划完整步骤，再交给执行器逐步执行，适合长任务和流程稳定场景。
Plan-and-Execute 的优点是全局性更强，方便用户确认和工程调度；缺点是前期计划可能和实际工具结果不一致，需要重规划机制。
生产中可以混合使用：先生成粗计划，再每一步用 ReAct 执行；如果观察结果偏离预期，就触发 replan。

第 45 题
Agent 开发中常见的系统安全风险有哪些，例如 Prompt Injection、沙箱逃逸和越权执行？应如何防范？
分类：Agent / LLM ｜ 难度：深入 ｜ 高频公司：未知
一句话：Agent 安全的核心是模型不可信，外部内容不可信，工具执行必须由系统硬控。
常见安全风险包括 Prompt Injection、工具越权、敏感数据泄露、恶意文件执行、沙箱逃逸、SSRF、SQL 注入、命令注入、供应链攻击和日志泄密。
Prompt Injection 是用户或外部文档诱导模型忽略系统规则。防范方式是区分指令和数据，外部内容只能作为不可信数据，不允许覆盖系统提示词和权限策略。
工具安全要做最小权限、白名单、参数校验、沙箱隔离、网络限制、文件路径限制、敏感操作确认和审计日志。
还要有输出安全：脱敏、权限过滤、引用来源、禁止泄露系统提示词和密钥。生产中不能让模型直接执行 shell 或访问任意网络。

第 46 题
Agent 耗时过长时，工程侧和基座侧分别有哪些优化手段？
分类：Agent / LLM ｜ 难度：进阶 ｜ 高频公司：未知

第 47 题
请介绍 MCP（Model Context Protocol）的作用、原理，以及你是否有过相关实践？
分类：MCP ｜ 难度：进阶 ｜ 高频公司：腾讯、蚂蚁、字节
一句话：MCP 是 AI 应用连接外部工具和数据源的标准协议层。
MCP 是一种让 AI 应用标准化连接外部系统的协议。它把外部能力抽象成 tools、resources、prompts 等，让模型应用不需要为每个系统单独写一套私有适配。
原理上，MCP 包含客户端和服务端。服务端暴露工具、资源和提示模板，客户端发现能力并把可用工具提供给模型。当模型选择工具时，客户端通过协议调用 MCP Server，再把结果返回给模型。
它的价值是降低集成成本、统一权限边界、复用工具生态。比如文件系统、数据库、Git、监控平台、搜索服务都可以作为 MCP Server 接入。
实践回答可以说：我会把内部查询指标、文档检索、工单系统、日志分析等能力封装成 MCP 工具，由 Agent Runtime 控制权限、审计和调用流程。

第 48 题
如何设计 Agent 的流式输出，以提升用户体验，尤其是在包含工具调用和多轮大模型交互的场景下？
分类：Agent ｜ 难度：进阶 ｜ 高频公司：未知
一句话：好的流式输出要让用户看到进度、证据和阶段结果，而不是只等最终答案。
Agent 流式输出不只是把最终答案逐字吐出来，还要让用户知道系统正在做什么。尤其多轮工具调用时，如果长时间无反馈，用户会以为卡住。
可以分层输出：先返回任务已接收，再输出阶段状态，比如“正在检索文档”“正在查询指标”“正在分析结果”，最后流式输出结论。
内部思考链不建议完整暴露，但可以暴露可解释的行动摘要和工具状态。比如展示调用了哪个只读工具、查询了什么时间范围、是否成功。
工程上可以用事件流：message_delta、tool_start、tool_result、step_summary、final_answer、error。前端根据事件类型渲染不同 UI。

第 49 题
Agent 长期记忆的 FIFO 淘汰策略有哪些问题，如何优化？
分类：Agent ｜ 难度：进阶 ｜ 高频公司：腾讯
一句话：长期记忆不能先进先出，要按价值、时效和可信度淘汰。
FIFO 只按写入时间淘汰，问题是可能删掉仍然重要的长期偏好或关键经验，也可能保留很多新但无价值的信息。
长期记忆淘汰应该综合重要性、最近使用时间、访问频率、置信度、用户确认、业务有效期和敏感级别。比如用户长期偏好比一次闲聊更重要。
可以设计 memory score：importance、recency、frequency、confidence、expiration 多因子加权。低分记忆先归档或删除，高价值记忆保留。
还要支持合并和更新，而不是重复存多条。比如用户岗位目标变化，应更新旧记忆状态，而不是简单追加。

第 50 题
什么是 Self-Reflection 机制？它在代码生成或故障排查 Agent 中如何应用？
分类：Agent ｜ 难度：进阶 ｜ 高频公司：未知
一句话：Self-Reflection 是让 Agent 有审核闭环，用证据检查自己有没有做对。
Self-Reflection 是让 Agent 在执行后对自己的结果进行检查、评价和修正。它不是单纯让模型再想一遍，而是基于明确标准检查输出是否满足目标。
在代码生成中，Reflection 可以检查编译错误、测试失败、接口兼容性、空指针、并发风险、是否改错文件。最好结合真实工具结果，比如编译器、单测、静态扫描，而不是只靠模型主观判断。
在故障排查中，Reflection 可以检查结论是否有证据支撑、是否遗漏关键指标、是否把相关性误判为因果、是否需要继续查询日志或监控。
工程上可以设置 Reviewer Agent 或 Critic 步骤，对执行结果打分并给出修复建议；但要限制反思轮数，避免无限自我修正。
