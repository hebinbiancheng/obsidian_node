# Harness Engineering 在硅谷爆火

> 原文：[Harness Engineering在硅谷爆火，一文带你搞懂！](https://mp.weixin.qq.com/s/1-bPS2omne-tncLTPm5N7A)  
> 作者：Datawhale  
> 整理时间：2026-06-11

## 一句话总结

Harness Engineering 的核心不是“换一个更聪明的模型”，而是为 AI Agent 搭建可持续工作的环境：**清晰的工作台、可用的工具、明确的验收标准、持续反馈机制和可观测系统**。模型提供智能，Harness 让智能真正变成可交付能力。

## 核心结论

- `Agent = Model + Harness`：模型只是智能来源，Harness 决定这个智能能否持续、稳定、可验证地工作。
- 大厂实践的关键变化是：工程师从“直接写代码”转向“设计环境、设定标准、建立反馈”。
- 不要把所有规则、背景、规范都塞进一个超长提示词或一个大文档；更好的方式是建立结构化文件系统，让 AI 在需要时读取对应资料。
- AI 自评通常不可靠，需要拆分“生成者”和“评估者”，或提供可执行的检查工具和明确评分标准。
- 长上下文不是万能解法；重要知识应该落到文件系统、文档、日志和可检索资料中，而不是全部塞进上下文窗口。
- 普通人也可以实践 Harness Engineering：从一个高频重复任务开始，逐步整理工作台、定义验收单、建立返工机制。

## 1. 什么是 Harness Engineering

文章把 Harness Engineering 解释为“给 AI 搭环境”。

类比来说，模型像一匹马，本身有力量；但没有马鞍、缰绳、马镫，人很难稳定驾驭它。Harness 就是给模型配上的这套“装备”。

在 Agent 场景中，Harness 通常包括：

- 系统提示词；
- 工具调用能力；
- 文件系统；
- 沙箱环境；
- 编排逻辑；
- 日志、指标、可观测性；
- Lint、测试、评估器等检查机制；
- 可读的项目文档和工作规范。

可以用三个问题理解 Harness：

1. **AI 在哪里干活？**  
   例如代码仓库、文件系统、文档目录、浏览器、沙箱。

2. **AI 用什么干活？**  
   例如 bash、测试命令、浏览器 DevTools、日志查询、代码执行工具。

3. **AI 怎么知道自己干得对不对？**  
   例如测试、lint、构建、验收标准、评估器、人工反馈。

如果这三个问题没有解决，AI 更像聊天工具；如果解决得好，AI 才更像长期合作的工程团队成员。

## 2. OpenAI 的实践：给 AI 一个整理好的工作台

文章提到 OpenAI 的一个实践：一开始他们把大量信息都塞进 `AGENTS.md`，包括系统说明、架构规范、代码风格和注意事项。结果 AI 被信息淹没，效果反而下降。

后来他们做了调整：

- 把 `AGENTS.md` 精简成约 100 行，主要作为“目录”；
- 真正的内容放进结构化的 `docs/` 目录；
- 设计文档、架构文档、计划文档分开存放；
- AI 需要什么信息时，再去对应位置读取。

这里的关键洞察是：

> AI 不需要一开始知道所有事情，而需要在正确时机拿到正确资料。

### 对应的工程启示

对于自己的项目，不建议把所有说明都写进一个巨大提示词。更推荐：

```text
项目根目录
├── AGENTS.md / CLAUDE.md        # 入口说明：告诉 AI 去哪里找什么
├── docs/
│   ├── architecture/            # 架构说明
│   ├── decisions/               # 技术决策记录
│   ├── workflows/               # 常用流程
│   ├── testing/                 # 测试规范
│   └── troubleshooting/         # 常见问题与排查
└── scripts/                     # 可重复执行的检查和工具脚本
```

这也是一种“可检索的工作台”。

## 3. 工具与可观测性：让 AI 能看到结果

文章强调，OpenAI 不只是给 AI 文件系统，还给 AI 配了工具：

- bash；
- 代码执行；
- 沙箱环境；
- 日志；
- 指标；
- UI 界面；
- Chrome DevTools 等浏览器验证能力。

这样 AI 不只是“写代码”，还可以：

1. 运行代码；
2. 看到错误；
3. 读取日志；
4. 分析指标；
5. 修改代码；
6. 再次验证。

这形成了一个闭环。

### Linter 的受众变化

文章里有一个重要细节：团队会重写 Linter 错误信息，让 AI 更容易理解“哪里错了、该怎么改”。

这说明 Harness Engineering 下，很多工具的使用者不再只是人类，也包括 AI。于是工具输出要更结构化、更明确、更可操作。

## 4. Anthropic 的实践：把生成和评估拆开

文章提到 Anthropic 的思路：AI 对自己的工作评价往往不可靠，尤其是设计、创意、前端体验这类任务。

解决方式是拆成两个角色：

- **生成器**：负责生成页面或应用。
- **评估器**：拿到工具后真实操作页面，点击按钮、填写表单、截图检查，然后根据标准给反馈。

评估维度包括：

| 维度 | 关注点 |
|---|---|
| 设计质量 | 视觉是否精致、层次是否清晰 |
| 原创性 | 是否有新意，而不是模板化输出 |
| 工艺 | 细节是否完整，交互是否顺滑 |
| 功能性 | 是否真的可用，核心功能是否跑通 |

生成器根据评估器反馈不断迭代，可能循环 5 到 15 轮。

### 核心启示

不要让同一个 AI 同时“创作”和“给自己打高分”。更可靠的方式是：

- 明确评估标准；
- 提供可操作环境；
- 让评估过程基于真实输出，而不是主观自夸；
- 把反馈重新输入生成过程。

## 5. 架构规则：用约束防止 AI 写乱

文章还提到 OpenAI 使用严格架构规则来限制代码依赖方向，例如：

```text
Types → Config → Repo → Service → Runtime → UI
```

如果代码违反依赖方向，就会被自动拦截。

这说明 Harness 不是只靠提示词约束 AI，而是要把关键约束变成：

- 自动化检查；
- 依赖规则；
- Lint 规则；
- 测试；
- CI Gate；
- 代码生成边界。

### 对自己的启示

对于长期项目，应该把架构原则写成 AI 可以读取、工具可以验证的形式。例如：

- 哪些目录不能互相依赖；
- 业务层不能直接访问 UI；
- API 层不能绕过 Service；
- 数据库访问只能走 Repository；
- 公共类型放在哪里；
- 测试文件如何命名。

越能被工具检查，越不依赖 AI 自觉。

## 6. 长上下文不是万能解法

文章指出，即使模型支持很长的上下文，性能也可能在较大 token 数后衰减。

因此，正确思路不是把所有资料都塞进上下文，而是：

- 把长期知识写入文件系统；
- 保持文档可检索；
- 需要时再读取；
- 定期清理过时文档；
- 建立文档维护 Agent 或文档巡检机制。

这与 Obsidian 的知识库思想非常一致：

- 不追求一次性塞满上下文；
- 追求知识稳定落地；
- 通过目录、标签、双链、索引让 AI 能找到正确资料。

## 7. 普通人如何实践 Harness Engineering

文章强调 Harness Engineering 不只是大厂实践，普通人也可以迁移。

核心转变是：

> 不再把 AI 当一次性聊天工具，而是给它工作手册、工具权限和验收标准。

### 7.1 整理 AI 的工作台

可以从以下方面开始：

- 建立清晰文件夹结构；
- 统一命名规范；
- 把参考资料、模板、规范放在固定位置；
- 告诉 AI 背景资料、模板、项目规范分别在哪里；
- 不把所有信息塞进一个文档，而是分层组织。

### 7.2 设定验收单

不要只说“写一篇文章”“做一个页面”“修一下 bug”，而要定义完成标准。

例如：

| 模糊任务 | 更好的验收标准 |
|---|---|
| 写一篇文章 | 3000 字、3 个案例、至少 2 个数据来源、有结论和行动项 |
| 做一个页面 | 支持桌面和移动端、核心按钮可点击、表单校验可见、无控制台错误 |
| 修一个 bug | 有复现步骤、有测试或验证命令、修复后原功能不回归 |
| 整理资料 | 有摘要、关键观点、标签、来源、后续行动 |

### 7.3 建立返工机制

AI 出错时，不要只责怪模型，而要问：

- 是否缺少背景资料？
- 是否缺少示例？
- 是否缺少验收标准？
- 是否缺少工具反馈？
- 是否缺少中间检查点？

每发现一个重复错误，就把解决办法固化进 Harness：文档、模板、规则、测试、脚本或检查清单。

## 8. 三周实践计划

文章给出的实践路径可以整理成三周计划：

### 第一周：整理文件结构和规范

目标：让 AI 知道去哪找东西。

可执行动作：

- 建立 `docs/`、`templates/`、`scripts/`、`examples/` 等目录；
- 写一个简短入口文档，例如 `AGENTS.md` / `CLAUDE.md`；
- 把常用规则从聊天记录迁移到文件；
- 给 AI 明确说明各目录用途。

### 第二周：设定验收标准

目标：让 AI 知道什么叫“做完”。

可执行动作：

- 为高频任务写验收清单；
- 为代码任务补充测试 / lint / build 命令；
- 为内容任务补充字数、结构、案例、来源要求；
- 为 UI 任务补充截图、交互和浏览器验证要求。

### 第三周：建立反馈机制

目标：让 AI 能自我检查和修正。

可执行动作：

- 将常见错误写入 troubleshooting；
- 建立检查脚本或命令；
- 对复杂任务设置阶段性验收点；
- 让 AI 在最终回复中报告验证结果和未验证风险。

## 10. 我的理解

这篇文章最重要的观点是：**AI 能力的瓶颈正在从模型本身，转向模型所处的工作环境。**

同样的 Claude、GPT 或 Codex：

- 没有 Harness 时，它像一个聪明但健忘、看不见结果、没有工作标准的实习生；
- 有 Harness 时，它像一个有工作台、有工具、有规范、有验收、有反馈的团队成员。

因此，未来高效使用 AI 的关键能力不是“会写更长 prompt”，而是会设计：

- 信息结构；
- 工具接口；
- 验收标准；
- 反馈回路；
- 约束系统；
- 可持续更新的知识库。
