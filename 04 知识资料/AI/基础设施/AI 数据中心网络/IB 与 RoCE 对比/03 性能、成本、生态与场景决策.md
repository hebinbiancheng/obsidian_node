# 性能、成本、生态与场景决策

## 1. 性能对比

两个原文都强调：IB 和 RoCEv2 的端口带宽已经可以同处 400G/800G 级别，差异更多体现在延迟、长尾稳定性、拥塞控制和整体利用率。

| 指标 | InfiniBand | RoCEv2 |
|---|---|---|
| 端到端延迟 | 通常 < 1μs | 常见约 1.5-5μs，拥塞时更高 |
| 单跳交换时延 | 文章二给出 100ns-300ns 级 | 文章二给出 400ns-1μs 左右 |
| 带宽 | 200G/400G/800G 均可 | 200G/400G/800G 均可 |
| 带宽利用率 | 常见可达 90%-95%+ | 调优好可接近，但更依赖哈希和拥塞控制 |
| 长尾稳定性 | 更强 | 更容易被拥塞、PFC、ECMP 哈希影响 |

## 2. 成本对比

文章一给出一个 512 GPU 集群的估算模型：IB 三年总成本约 4.61M 美元，以太网 400G 约 2.37M 美元，相差约 2.24M 美元。

这类数字不应机械套用到所有项目，但说明一个关键事实：

> 在中等规模集群里，RoCE 省下来的网络预算，可能足以购买更多 GPU 或覆盖一部分运维投入。

因此，财务选型要问：

- IB 带来的训练加速是否能覆盖网络溢价？
- 项目是否真正处在“训练窗口就是生命线”的阶段？
- 团队是否有能力把 RoCE 调到稳定状态？
- 供应链绑定风险是否可接受？

## 3. 生态对比

| 维度 | InfiniBand | RoCEv2 |
|---|---|---|
| 供应商 | NVIDIA/Mellanox 主导 | Broadcom、Cisco、Arista、Intel、AMD 等多生态 |
| 工具链 | UFM 等成熟专有工具 | SONiC、Telemetry、BGP/SDN 生态快速发展 |
| 互操作 | 更强调同一生态套件 | 更开放，可混合供应商 |
| 人才 | 专门人才较少 | 以太网人才基数大 |
| 风险 | 供应商绑定 | 调优复杂、质量依赖工程能力 |

## 4. 典型用户与行业路线

原文一列举了若干行业路线：

- Meta：大规模投入 RoCE，同时也运行 IB 集群；
- OpenAI：GPT-4 训练与微软 Azure IB 集群有关，体现对训练速度和确定性的偏好；
- Google：使用自研 Jupiter 网络，不是普通以太网；
- 阿里巴巴：训练与推理场景可能混合使用不同网络；
- 传统 HPC：IB 仍然有很深的软件和行业基础。

这些案例说明：大型组织不是简单“二选一”，而是按训练、推理、存储、成本、供应链进行组合。

## 5. 场景决策表

| 场景 | 推荐路线 | 判断依据 |
|---|---|---|
| 万卡级预训练 | IB 优先 | 通信效率和长尾稳定性直接影响训练窗口 |
| 千卡级以内、成本敏感 | RoCEv2 优先 | 成本优势明显，性能可通过工程调优逼近 |
| 国产算力芯片集群 | RoCEv2 常见 | 以太网适配和生态开放度更有优势 |
| NVMe-oF / 高性能存储 | RoCEv2 | 流量相对可控，性价比高 |
| 科学计算、强耦合 HPC | IB | 低延迟、MPI/HPC 生态成熟 |
| 推理集群 | RoCEv2 / Spectrum-X 类以太网方案 | 推理更看重规模、成本和通用数据中心整合 |

## 6. 财务视角的简化公式

```text
是否选择 IB = 训练加速收益 + 稳定性收益 + 团队熟练度收益
              > 网络硬件溢价 + 生态绑定成本 + 人才成本

是否选择 RoCE = 成本节省 + 供应链开放 + 统一以太网收益
                > 调优成本 + 长尾风险 + 排障复杂度
```
