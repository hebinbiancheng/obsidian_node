# K8s 生产故障全景图：8 大场景速查

> 原文：[微信文章](https://mp.weixin.qq.com/s/TVHjV9-9FqGzRyfemGwvkg) · 2026-07-19

## 一句话总结

运维老手的成长不是记了多少命令，而是看到 `Pod Evicted` 秒联想「节点内存压力 + 没设 requests」，看到 DNS 失败能区分「CoreDNS 挂了」还是「ndots 放大」。8 大故障域 + 五大通用防呆原则。

---

## 一、8 大故障域全景

| 故障域 | 高频故障 | 一句话方案 |
|--------|---------|-----------|
| 资源调度 | OOMKilled / Throttling / Evicted / Pending | 调 requests/limits，加节点，去 CPU limit |
| 网络 | DNS 失败 / Service 不通 / Ingress 5xx / 连接 reset | 查 CoreDNS / 标签 / readiness / conntrack |
| 存储 | PVC Pending / 挂载失败 / PV 误删 | 配 StorageClass，设 Retain，核对访问模式 |
| 发布配置 | 滚动卡住 / ConfigMap 不生效 / ImagePull / CrashLoop | 查 readiness、热更新机制、镜像仓库 |
| 控制面 | etcd 压力大 / API 超时 / kubelet 异常 / 证书过期 | defrag、减负、看运行时、续证书 |
| 可观测性 | 日志丢失 / 监控断点 / 告警风暴 | 保采集器在线，加分组抑制 |
| 高可用灾难 | 节点宕机 / 级联故障 / 脑裂 / 雪崩 | 多副本 + PDB + 降级 + 限流 |
| 安全权限 | RBAC 拒绝 / 提权限制 / 镜像漏洞 | 最小权限，OPA 准入，扫镜像 |

---

## 二、资源调度域（80% 故障来源）

| 现象 | 根因 | 处理 |
|------|------|------|
| OOMKilled | 内存超 limit，oom killer 杀容器 | 调大 memory limit；Java 用 MaxRAMPercentage |
| CPU Throttling | cpu limit 太低，cfs quota 节流 | 去 cpu limit 或调高 |
| Evicted | 节点内存/磁盘压力触发驱逐 | 加节点、合理设 requests、调 eviction 阈值 |
| Pending | 资源不足 / 亲和不匹配 / taint | kubectl describe 看 Events；加节点或减 requests |

> **铁律**：requests 决定调度，limits 决定被杀。只设 limit 不设 requests 是新手最常见的错——调度器以为节点很空，全堆一起，节点一紧张集体 Evicted。

---

## 三、网络域

| 现象 | 根因 | 处理 |
|------|------|------|
| DNS 失败 | CoreDNS 挂 / ndots=5 放大 / 重启缓存清空 | 查 CoreDNS Pod；缩短 ndots；用全限定域名 |
| Service 不通 | selector 与 Pod label 不匹配 / endpoints 空 | kubectl get endpoints 核对 |
| Ingress 5xx | 后端 Pod readiness 不绿 / 超时 / 连接数满 | 查 readinessProbe 与 upstream 超时 |
| 连接 reset | conntrack 表满 / TIME_WAIT 堆积 | 调大 conntrack 上限 |

> **真实坑**：kubelet 重启后容器 DNS 缓存清空，重解析期间若 CoreDNS 恰在滚动更新，会出现 5-15 秒 DNS 失败窗口。

---

## 四、存储域

| 现象 | 根因 | 处理 |
|------|------|------|
| PVC Pending | 无匹配 StorageClass | 确认 StorageClass 存在且 default 正确 |
| 挂载失败 | RWO 多副本冲突 / fsGroup 权限 | 核对访问模式；设 fsGroup |
| **PV 误删** | 回收策略 default 是 **Delete** | ⚠️ 关键数据 PV 设 Retain + Velero 备份 |

---

## 五、发布与配置域

| 现象 | 根因 | 处理 |
|------|------|------|
| 滚动卡住 | readiness 不绿 / maxUnavailable=0 且资源不够 | pause 后回滚；放宽发布参数 |
| ConfigMap 不生效 | subPath 不热更新 | 改 CM 后滚动一次；用 reload 机制 |
| ImagePullBackOff | 私仓无 imagePullSecret / 仓库不可达 | 配 imagePullSecret；查仓库 DNS/凭证 |
| CrashLoopBackOff | 依赖未起 / 探测配错 / 配置写错 | kubectl logs 看启动报错 |

---

## 六、控制面域

| 现象 | 根因 | 处理 |
|------|------|------|
| etcd 压力大 | 事件量爆炸 / 未压缩 / 碎片 | etcdctl defrag；限事件保留；加配额 |
| API 超时 | 负载高 / watch 过多 / 证书问题 | 减负、扩 API 副本、查证书 |
| kubelet 异常 | 容器运行时挂 / cgroup 不一致 | 查 containerd；对齐 cgroup v1/v2 |
| **证书过期** | kubeadm 证书默认 1 年 | kubeadm certs renew；设自动续期 |

> kubelet 重启不会杀 Pod——但运行时挂了 Pod 变空壳（Status Running 无进程），Service 还在往空壳导流量，要等下次心跳才发现。

---

## 七、五大通用防呆原则

1. **requests 必设**：只设 limit 不设 requests 是灾难之源
2. **多副本 + PDB**：单副本就是定时炸弹，PDB 防一次驱逐干光
3. **监控隔离部署**：监控和业务同节点 = 盲人救火
4. **分批操作**：批量重启/驱逐 = 手动制造连接风暴
5. **回滚预案在手**：发版前确认能 30 秒 undo，别现场翻文档
