# Agnes 使用策略、风险与工作流建议

## 1. 原文的核心策略

作者建议的合理用法是：

```text
Claude Opus / 高能力模型：任务规划、复杂判断、关键决策
Agnes 免费模型：执行、试错、生成多个候选、低风险任务
```

也就是说，不要把 Agnes 当成完全替代品，而是放到工作流中的执行层。

## 2. 推荐工作流

### 2.1 代码项目分析

```text
高能力模型制定分析框架
  ↓
Agnes 扫描目录、生成结构文档
  ↓
人工检查关键路径
  ↓
高能力模型复核架构判断
```

### 2.2 前端 Demo 生成

```text
明确页面需求
  ↓
Agnes 生成静态 HTML/CSS Demo
  ↓
人工浏览器验证
  ↓
再交给高能力模型做可维护性优化
```

### 2.3 图像 / 视频素材生成

```text
明确风格和约束
  ↓
Agnes 批量生成候选
  ↓
人工挑选
  ↓
必要时二次编辑或高规格重跑
```

## 3. 适合 Agnes 的任务

| 任务 | 原因 |
|---|---|
| 多版本草稿 | 免费试错成本低 |
| 结构化文档初稿 | 对准确性有人工复核空间 |
| 静态页面 demo | 易验证、易迭代 |
| 架构图草稿 | prompt 明确时产出可用性较高 |
| 短视频候选 | 视频本身试错成本高，免费 API 有优势 |
| 项目初步阅读 | 可作为第一轮扫描，不直接决定架构 |

## 4. 需要谨慎的任务

| 任务 | 风险 |
|---|---|
| 核心代码自动修改 | 可能引入隐蔽 bug |
| 生产素材批量输出 | 质量不稳定，需要人工筛选 |
| 长任务无人值守 | 免费服务可能卡顿或不稳定 |
| 高事实准确性内容 | 需要额外查证 |
| 高安全/合规任务 | 不应直接外发敏感代码或数据 |

## 5. 使用原则

### 原则 1：先小任务试水

不要一上来把整个核心仓库或生产任务交给 Agnes。先切一小块任务测试稳定性。

### 原则 2：把免费当成试错优势

免费 API 最大价值是降低试错成本。适合多跑几版、挑选最好结果。

### 原则 3：关键任务保留人工验收

无论文本、图片还是视频，关键结果都要人工检查。

### 原则 4：规划和执行分离

复杂任务建议先让高能力模型做任务拆解，再让 Agnes 执行明确子任务。

### 原则 5：预留时间成本

免费服务可能在高峰期卡顿，生成结果也可能需要多次重跑。要提前预留时间。

## 6. 我的理解

这篇文章真正值得沉淀的不是“某个免费 API 很香”，而是一种 AI 工具组合思路：

> 高质量模型负责少量高价值思考，低成本模型负责大量执行和试错。

这和工程里的分层思想类似：

- 架构师负责方案；
- 工程师负责实现；
- 自动化工具负责重复任务；
- 评审机制负责质量兜底。

在 AI 工作流中，也应该按模型能力和成本做分层，而不是所有事情都交给最贵模型。
